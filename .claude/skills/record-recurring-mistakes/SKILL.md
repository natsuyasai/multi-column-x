# 再発ミスのCLAUDE.md追記（横断ルール）

アプリ開発フローの全フェーズを通じて、**同様のミスの指摘が複数回（2回以上）発生したら、必ず `CLAUDE.md` に再発防止ルールを追記する。**

レビュー指摘、Lint/型エラーの繰り返し、規約違反の再発などが対象。「次から気をつける」で終わらせず、ドキュメント化して恒久対策にする。

## 進め方

1. **検知**: 同種の指摘・修正が2回以上出たことに気づく（同一セッション内・複数セッションまたぎ問わない）。
2. **一般化**: 個別事象ではなく、再発を防ぐ一般的なルールとして言語化する。
3. **追記**: `CLAUDE.md` の適切な箇所（実装ガイドライン／作業手順／アーキテクチャ上の重要な制約 など）に、具体的な指示として追記する。
4. **共有**: 追記内容をユーザーに伝える。

## 追記の書き方

- 何を・なぜ・どうするかを具体的に。再現条件と正しい対応をセットで書く。
- 悪い例: 「import順に注意する」
- 良い例: 「import順は外部パッケージ → `@/` エイリアス（internal）→ 相対パスの順。グループ間の空行は入れない（`newlines-between: "never"`）。`eslint --fix` で自動整列できる」

## このプロジェクトで特に再発しやすい論点

- **desktop / mobile の二重実装**: 同名コマンドでも `#[cfg(desktop)]` / `#[cfg(mobile)]` で実装が異なる。片方だけ直して再発しやすい。
- **serde のフィールド命名**: JS側 camelCase には `#[serde(rename = "...")]` が必須。
- **Android ProGuard keep ルール**: `MainActivity.kt` のメソッドシグネチャ変更時の keep ルール同期漏れ（リリースビルドでのみ顕在化）。
- **inject スクリプト**: `_src` を変更したら `npm run build:inject` が必要。ビルド済み `.js` の直接編集禁止。
- **ショートカット追加時の同期箇所**（既知の再発ポイント）。

## 判断の境界

- **CLAUDE.mdに追記する**: プロジェクト全体に効く規約・手順・繰り返す判断ミス
- **スキルを修正/新規作成する**: 手順が複雑で独立したワークフローになるもの
- **メモリに記録する**: コードからは読み取れない非自明な背景・経緯

## 禁止事項

- 同じ指摘が複数回出ているのに口頭の反省だけで済ませること
- 個別事象そのままで、一般化せずに追記すること
