# 質問フォーマットガイド

## 必須：すべての質問はこのフォーマットを使用すること

### ルール：チャットで質問を直接しない

**重要**：チャット内で直接質問してはいけません。すべての質問は専用の質問ファイルに配置してください。

### 質問ファイルのフォーマット

#### ファイル命名規則

- 説明的な名前を使用：`{フェーズ名}-questions.md`
- 例：
  - `classification-questions.md`
  - `requirements-questions.md`
  - `story-planning-questions.md`
  - `design-questions.md`

#### 質問の構造

すべての質問には意味のあるオプションと最後のオプションとして「その他」を含める必要があります：

```markdown
## 質問 [番号]
[明確で具体的な質問文]

A) [1番目の意味のあるオプション]
B) [2番目の意味のあるオプション]
[...必要に応じて追加オプション...]
X) その他（以下の[Answer]:タグの後に記述してください）

[Answer]:
```

**重要**：

- 「その他」はすべての質問の最後のオプションとして**必須**です
- 意味のあるオプションのみを含める——スロットを埋めるためにオプションを作り上げないこと
- 意味のある選択肢の数は適宜調整する（最低2つ + その他）

E) その他（以下の[Answer]:タグの後に記述してください）

A) Webアプリケーション
B) モバイルアプリケーション
C) WebとモバイルI両方
D) その他（以下の[Answer]:タグの後に記述してください）

### ユーザー回答フォーマット

ユーザーは[Answer]:タグの後に文字の選択肢を記入して回答します：

```markdown
## 質問 1
プライマリユーザー認証方式は何ですか？

A) ユーザー名とパスワード
B) ソーシャルメディアログイン（Google、Facebook）
C) シングルサインオン（SSO）
D) 多要素認証

[Answer]: C
```

### ユーザー回答の読み方

ユーザーが完了を確認した後：

1. 質問ファイルを読む
2. [Answer]:タグの後の回答を抽出する
3. すべての質問に回答されているか検証する
4. 回答に基づいて分析を進める

### 複数選択のガイドライン

#### オプションの数

- 最小：2つの意味のあるオプション + 「その他」（A、B、C）
- 典型：3〜4つの意味のあるオプション + 「その他」（A、B、C、D、E）
- 最大：5つの意味のあるオプション + 「その他」（A、B、C、D、E、F）
- **重要**：スロットを埋めるためにオプションを作り上げないこと——意味のある選択肢のみを含める

#### オプションの品質

- 相互排他的なオプションにする
- 最も一般的なシナリオをカバーする
- 意味のある現実的なオプションのみを含める
- **最後のオプションとして必ず「その他」を含める**（必須）
- 具体的かつ明確にする
- **A、B、C、Dのスロットを埋めるためにオプションを作り上げないこと**

#### 良い例

```markdown
## 質問 5
使用するデータベース技術は何ですか？

A) リレーショナル（PostgreSQL、MySQL）
B) NoSQLドキュメント（MongoDB、DynamoDB）
C) NoSQLキーバリュー（Redis、Memcached）
D) グラフデータベース（Neo4j、Neptune）
E) その他（以下の[Answer]:タグの後に記述してください）

[Answer]:
```

#### 悪い例（避けるべき）

```markdown
## 質問 5
使用するデータベースは何ですか？

A) はい
B) いいえ
C) わかりません

[Answer]:
```

### ワークフローとの統合

#### ステップ1：質問ファイルの作成

```markdown
aidlc-docs/{フェーズ名}-questions.md にすべての質問を作成する
```

#### ステップ2：ユーザーへの通知

```
「{フェーズ名}-questions.mdに[X]個の質問を作成しました。
[Answer]:タグの後に文字の選択肢を記入して各質問に答えてください。
オプションがニーズに合わない場合は最後のオプション（その他）を選択して希望を記述してください。完了したらお知らせください。」
```

#### ステップ3：確認を待つ

ユーザーが「完了」、「done」、「finished」などと言うまで待つ。

#### ステップ4：読んで分析する

```
aidlc-docs/{フェーズ名}-questions.md を読む
すべての回答を抽出する
完全性を検証する
分析を進める
```

### エラー処理

#### 回答が欠けている場合

```
「質問[X]に回答がないことに気づきました。先に進む前にすべての質問の文字選択肢を使って回答を提供してください。」
```

#### 無効な回答の場合

回答が有効な文字選択肢でない場合：

```
「質問[X]の回答「[回答]」が無効です。
質問に提供された文字選択肢のみを使用してください。」
```

#### 曖昧な回答の場合

ユーザーが文字ではなく説明を提供した場合：

```
「質問[X]については、回答に最も合う文字選択肢を提供してください。
合うものがない場合は「その他」を選択して[Answer]:タグの後に説明を追加してください。」
```

### 矛盾と曖昧さの検出

**必須**：ユーザーの回答を読んだ後、矛盾と曖昧さを必ずチェックすること。

#### 矛盾の検出

論理的に一貫性のない回答を探す：

- スコープの不一致：「バグ修正」だが「コードベース全体に影響」
- リスクの不一致：「低リスク」だが「破壊的な変更」
- タイムラインの不一致：「クイックフィックス」だが「複数のサブシステム」
- 影響の不一致：「単一コンポーネント」だが「重大なアーキテクチャの変更」

#### 曖昧さの検出

不明確またはボーダーラインな回答を探す：

- 複数の分類に当てはまる可能性のある回答
- 具体性を欠く回答
- 複数の質問にわたる相反する指標

#### 明確化質問の作成

矛盾または曖昧さが検出された場合：

1. **明確化ファイルを作成**：`{フェーズ名}-clarification-questions.md`
2. **問題を説明**：何の矛盾/曖昧さが検出されたかを明確に述べる
3. **的を絞った質問をする**：問題を解決するために複数選択形式を使用する
4. **元の質問を参照**：どの質問が相反する回答を持っていたかを示す

**例**：

```markdown
# [フェーズ名] 明確化質問

回答に矛盾があり、明確化が必要です：

## 矛盾 1: [簡単な説明]
「[回答A]」（Q[X]:[文字]）とも「[回答B]」（Q[Y]:[文字]）とも示されています。
[説明]のため、これらの回答は矛盾しています。

### 明確化質問 1
[矛盾を解決するための具体的な質問]

A) [最初の回答に向けて解決するオプション]
B) [2番目の回答に向けて解決するオプション]
C) [中間地点を提供するオプション]
D) [質問を再定義するオプション]

[Answer]:

## 曖昧さ 1: [簡単な説明]
Q[X]への回答（「[回答]」）は[説明]のため曖昧です。

### 明確化質問 2
[曖昧さを明確にするための具体的な質問]

A) [明確なオプション1]
B) [明確なオプション2]
C) [明確なオプション3]
D) [明確なオプション4]

[Answer]:
```

#### 明確化のワークフロー

1. **検出**：すべての回答に矛盾/曖昧さがないか分析する
2. **作成**：問題が見つかった場合は明確化質問ファイルを生成する
3. **通知**：問題と明確化ファイルについてユーザーに伝える
4. **待機**：ユーザーが明確化を提供するまで進まない
5. **再検証**：明確化後に一貫性を再度確認する
6. **進行**：すべての矛盾が解決された場合のみ前進する

#### ユーザーへのメッセージ例

```
「回答に2つの矛盾が見つかりました：

1. バグ修正スコープとコードベースへの影響（Q1対Q2）
2. 低リスクと破壊的変更（Q7対Q4）

これらを解決するための2つの質問を含むclassification-clarification-questions.mdを作成しました。
分類を進める前にこれらの明確化質問に答えてください。」
```

### ベストプラクティス

1. **具体的に**：質問は明確で曖昧さがないようにする
2. **包括的に**：必要な情報をすべてカバーする
3. **簡潔に**：質問を1つのトピックに集中させる
4. **実用的に**：オプションは現実的で実行可能なものにする
5. **一貫性を保つ**：すべての質問ファイルで同じフォーマットを使用する

## まとめ

**覚えておいてください**：

- ✅ 常に質問ファイルを作成する
- ✅ 常に複数選択形式を使用する
- ✅ **常に「その他」を最後のオプションとして含める（必須）**
- ✅ 意味のあるオプションのみを含める——スロットを埋めるためにオプションを作り上げない
- ✅ 常に[Answer]:タグを使用する
- ✅ 常にユーザーの完了を待つ
- ✅ 常に矛盾がないか回答を検証する
- ✅ 必要な場合は常に明確化ファイルを作成する
- ✅ 常に先に進む前に矛盾を解決する
- ❌ チャットで質問しない
- ❌ A、B、C、Dを埋めるためだけにオプションを作り上げない
- ❌ 回答なしで先に進まない
- ❌ 未解決の矛盾のまま先に進まない
- ❌ 曖昧な回答について推測しない
