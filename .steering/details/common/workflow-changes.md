# ワークフロー中の変更とフェーズ管理

## 概要

ユーザーはワークフロー中に実行計画やフェーズの実行に対して変更を要求する場合があります。このドキュメントは、これらのリクエストを安全かつ効果的に処理するための指針を提供します。

---

## ワークフロー中の変更の種類

### 1. スキップされたフェーズの追加

**シナリオ**：ユーザーが元々スキップされたフェーズを追加したい

**例**：「やはり、あのステージをスキップしたけどユーザーストーリーを追加したい」

**処理方法**：

1. **リクエストを確認**：「ユーザーストーリーステージを追加したいですね。ユーザーストーリーとペルソナが作成されます。確認しますか？」
2. **依存関係を確認**：すべての前提フェーズが完了していることを確認する
3. **実行計画を更新**：理由とともに`execution-plan.md`にフェーズを追加する
4. **状態を更新**：`aidlc-state.md`でフェーズを「保留（PENDING）」としてマークする
5. **フェーズを実行**：通常のフェーズ実行プロセスに従う
6. **変更をログ記録**：タイムスタンプと理由とともに`audit.md`に文書化する

**考慮事項**：

- 新しい成果物から恩恵を受ける可能性のある後続フェーズの更新が必要な場合がある
- 新しい情報を組み込むために既存の成果物の修正が必要な場合がある
- タイムラインが延長される

---

### 2. 計画されたフェーズのスキップ

**シナリオ**：ユーザーが実行が計画されていたフェーズをスキップしたい

**例**：「NFR設計ステージを今はスキップしよう」

**処理方法**：

1. **リクエストを確認**：「NFR設計をスキップしたいですね。これはNFRパターンや論理コンポーネントが組み込まれないことを意味します。確認しますか？」
2. **影響について警告する**：何が欠けるか、潜在的な結果を説明する
3. **明示的な確認を得る**：ユーザーは影響を理解していることを明示的に確認する必要がある
4. **実行計画を更新**：理由とともにフェーズを「スキップ済み（SKIPPED）」としてマークする
5. **状態を更新**：`aidlc-state.md`でフェーズを「スキップ済み（SKIPPED）」としてマークする
6. **後続フェーズを調整**：後のフェーズで手動セットアップが必要な場合があることを注記する
7. **変更をログ記録**：タイムスタンプと理由とともに`audit.md`に文書化する

**考慮事項**：

- 後のフェーズが失敗するか手動介入が必要になる可能性がある
- ユーザーが欠けている成果物の責任を受け入れる
- 後で必要に応じて追加できる

---

### 3. 現在のステージの再開

**シナリオ**：ユーザーが現在のステージの結果に不満でやり直したい

**例**：「このユーザーストーリーが気に入らない。最初からやり直せますか？」

**処理方法**：

1. **懸念を理解する**：「ストーリーの何を変えたいですか？」
2. **オプションを提示する**：
   - **オプションA**：既存の成果物を修正する（より速い、一部の作業を保存）
   - **オプションB**：完全な再開（白紙の状態、より多くの時間）
3. **再開を選択した場合**：
   - 既存の成果物をアーカイブ：`{成果物}.backup.{タイムスタンプ}`
   - 計画ファイルのステージチェックボックスをリセット
   - `aidlc-state.md`でステージを「進行中（IN PROGRESS）」としてマーク
   - ステージの完了ステータスをクリア
   - 最初から再実行する
4. **変更をログ記録**：再開の理由と変更点を文書化する

**考慮事項**：

- 既存の作業が失われる（ただしバックアップされる）
- 依存するステージをやり直す必要がある場合がある
- タイムラインが延長される

---

### 4. 以前のステージの再開

**シナリオ**：ユーザーが完了したステージに戻ってやり直したい

**例**：「以前に行ったアーキテクチャの決定を変えたい」

**処理方法**：

1. **影響を評価する**：再開するステージに依存しているすべてのステージを特定する
2. **ユーザーに警告する**：「アプリケーション設計を再開すると、以下をやり直す必要があります：ユニットプランニング、ユニット生成、ユニットごとの設計（すべてのユニット）、コードプランニング、コード生成。確認しますか？」
3. **明示的な確認を得る**：ユーザーは完全な影響を理解する必要がある
4. **確認された場合**：
   - 影響を受けたすべての成果物をアーカイブする
   - `aidlc-state.md`で影響を受けたすべてのステージをリセット
   - 影響を受けたすべての計画ファイルのチェックボックスをクリア
   - 再開するステージに戻る
   - その時点から再実行する
5. **変更をログ記録**：完全な影響と再開の理由を文書化する

**考慮事項**：

- 大幅なやり直しが必要
- 依存するすべてのステージをやり直す必要がある
- タイムラインが大幅に延長される
- 再開より修正の方が良い場合があるかを検討する

---

### 5. ステージ深度の変更

**シナリオ**：ユーザーが現在または将来のステージの深度レベルを変更したい

**例**：「標準ではなく包括的な要件分析をやってみよう」

**処理方法**：

1. **リクエストを確認**：「要件分析を標準から包括的な深度に変更したいですね。より徹底的になりますが時間がかかります。確認しますか？」
2. **実行計画を更新**：`workflow-planning.md`で深度レベルを変更する
3. **アプローチを調整**：ステージの包括的な深度ガイドラインに従う
4. **見積もりを更新**：ユーザーに新しいタイムライン見積もりを通知する
5. **変更をログ記録**：深度の変更と理由を文書化する

**考慮事項**：

- より深い深度 = より多くの時間だがより良い品質
- 浅い深度 = より速いが詳細を見逃す可能性がある
- ステージ完了後ではなく、開始前または途中でのみ変更できる

---

### 6. ワークフローの一時停止

**シナリオ**：ユーザーが一時停止して後で再開する必要がある

**例**：「今は止めて明日続けます」

**処理方法**：

1. **現在のステップを完了**：可能であれば現在進行中のステップを終了する
2. **チェックボックスを更新**：完了したすべてのステップを[x]でマークする
3. **状態を更新**：`aidlc-state.md`が現在の状態を反映していることを確認する
4. **一時停止をログ記録**：`audit.md`に一時停止ポイントを文書化する
5. **再開手順を提供**：「戻ったときに既存のプロジェクトを検出して、[現在のフェーズ、現在のステップ]から継続するオファーをします」

**再開時**：

1. **既存のプロジェクトを検出**：`aidlc-state.md`を確認する
2. **コンテキストを読み込む**：完了したステージからすべての成果物を読み込む
3. **状態を表示**：現在のステージと次のステップを表示する
4. **オプションを提示する**：中断した箇所から継続するか以前の作業をレビューするか
5. **再開をログ記録**：`audit.md`に再開ポイントを文書化する

---

### 7. アーキテクチャの決定変更

**シナリオ**：ユーザーがモノリスからマイクロサービスへ（またはその逆に）変更したい

**例**：「実はモノリスではなくマイクロサービスにしよう」

**処理方法**：

1. **現在の進捗を評価する**：ワークフローがどこまで進んでいるかを判断する
2. **影響を説明する**：
   - ユニットプランニング前の場合：影響最小、決定を更新するだけ
   - ユニットプランニング後の場合：ユニットプランニング、ユニット生成、すべてのユニットごとの設計をやり直す必要がある
   - コード生成後の場合：大幅なやり直しが必要
3. **アプローチを推薦する**：
   - ワークフローの早い段階：アプリケーション設計ステージから再開する
   - ワークフローの遅い段階：変更が再開と比較して実現可能かどうかを検討する
4. **確認を得る**：ユーザーは変更の完全なスコープを理解する必要がある
5. **変更を実行**：影響を受けたステージの再開手順に従う

**考慮事項**：

- アーキテクチャの変更は連鎖的な影響を持つ
- ワークフローの早い段階 = 変更がより簡単
- ワークフローの遅い段階 = コスト対便益を検討する

---

### 8. ユニットの追加/削除

**シナリオ**：ユニット生成後にユーザーがユニットを追加または削除したい

**例**：「Paymentユニットをの支払いと請求に分割する必要がある」

**処理方法**：

1. **影響を評価する**：どのユニットが設計/コードを完了しているかを判断する
2. **影響を説明する**：
   - ユニットの追加：新しいユニットのための完全な設計とコードが必要
   - ユニットの削除：機能を他のユニットに再分配する必要がある
   - ユニットの分割：結果として生じる両方のユニットの設計とコードをやり直す必要がある
3. **ユニット成果物を更新する**：
   - `unit-of-work.md`を修正する
   - `unit-of-work-dependency.md`を更新する
   - `unit-of-work-story-map.md`を修正する
4. **影響を受けたユニットをリセット**：影響を受けたユニットを再設計が必要としてマークする
5. **変更を実行**：影響を受けたユニットの通常のユニット設計とコードプロセスに従う

**考慮事項**：

- これらのユニットの後続ステージすべてに影響する
- 依存関係が変わる場合は他のユニットにも影響する可能性がある
- タイムラインへの影響は影響を受けるユニットの数に依存する

---

## 変更を処理するための一般的なガイドライン

### 変更を行う前に

1. **リクエストを理解する**：ユーザーが何を変えたいか、その理由について明確化の質問をする
2. **影響を評価する**：影響を受けるすべてのステージ、成果物、依存関係を特定する
3. **影響を説明する**：何をやり直す必要があるか、タイムラインへの影響を明確に伝える
4. **代替案を提示する**：再開より修正の方が良い場合もある
5. **明示的な確認を得る**：ユーザーは影響を理解して受け入れる必要がある

### 変更中

1. **既存の作業をアーカイブする**：破壊的な変更を行う前に必ずバックアップする
2. **すべてのトラッキングを更新する**：`aidlc-state.md`、計画ファイル、`audit.md`を同期して保つ
3. **進捗をコミュニケーションする**：何が起きているかをユーザーに通知し続ける
4. **変更を検証する**：変更がすべての成果物全体で一貫していることを確認する
5. **継続性をテストする**：変更後にワークフローがスムーズに継続できることを確認する

### 変更後

1. **一貫性を確認する**：すべての成果物が変更に合わせて調整されているか確認する
2. **ドキュメントを更新する**：すべての参照が更新されていることを確認する
3. **完全にログ記録する**：`audit.md`に完全な変更履歴を文書化する
4. **ユーザーに確認する**：変更がユーザーの期待を満たしているか確認する
5. **ワークフローを再開する**：新しい状態から通常の実行を継続する

## 変更リクエストの決定ツリー

```
ユーザーが変更をリクエスト
    |
    ├─ 現在のフェーズか？
    |   ├─ はい：現在のフェーズを修正または再開できる
    |   └─ いいえ：次の質問へ
    |
    ├─ 完了したフェーズか？
    |   ├─ はい：依存するフェーズへの影響を評価する
    |   |   ├─ 影響が低い：修正して依存するものを更新する
    |   |   └─ 影響が高い：そのフェーズからの再開を推奨する
    |   └─ いいえ：次の質問へ
    |
    ├─ スキップされたフェーズの追加か？
    |   ├─ はい：前提条件を確認し、計画に追加し、実行する
    |   └─ いいえ：次の質問へ
    |
    ├─ 計画されたフェーズのスキップか？
    |   ├─ はい：影響について警告し、確認を得て、スキップする
    |   └─ いいえ：次の質問へ
    |
    └─ 深度レベルの変更か？
        ├─ はい：計画を更新し、アプローチを調整する
        └─ いいえ：ユーザーにリクエストを明確にするよう求める
```

## ログ記録要件

### 変更リクエストのログフォーマット

```markdown
## 変更リクエスト - [フェーズ名]
**タイムスタンプ**：[ISOタイムスタンプ]
**リクエスト**：[ユーザーが変更したいこと]
**現在の状態**：[ワークフローの現在位置]
**影響の評価**：[何が影響を受けるか]
**ユーザーの確認**：[ユーザーの明示的な確認]
**実行されたアクション**：[何をしたか]
**影響を受けた成果物**：[変更/リセットされたファイルのリスト]

---
```

## ベストプラクティス

1. **常に確認する**：明示的なユーザーの確認なしに破壊的な変更を行わない
2. **影響を説明する**：ユーザーは決定する前に影響を理解する必要がある
3. **オプションを提示する**：変更を処理する複数の方法がある場合がある
4. **最初にアーカイブする**：破壊的な変更を行う前に必ずバックアップする
5. **すべてを更新する**：すべてのトラッキングファイルを同期して保つ
6. **徹底的にログ記録する**：監査証跡のためにすべての変更を文書化する
7. **変更後に検証する**：ワークフローがスムーズに継続できることを確認する
8. **柔軟であること**：ワークフローはユーザーのニーズに適応すべきで、硬直したプロセスを強制しない
