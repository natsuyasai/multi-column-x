# エラー処理と回復手順

## 一般的なエラー処理の原則

### エラーが発生した場合

1. **エラーを特定する**：何が問題になったかを明確に述べる
2. **影響を評価する**：エラーが阻止的なものか、回避策があるかを判断する
3. **コミュニケーションする**：エラーとオプションについてユーザーに通知する
4. **解決策を提供する**：解決または回避するための明確な手順を提供する
5. **文書化する**：エラーと解決策を`audit.md`にログ記録する

### エラー重大度レベル

**Critical（クリティカル）**：ワークフローを継続できない

- 必要なファイルや成果物が欠けている
- 処理できない無効なユーザー入力
- ファイル操作を妨げるシステムエラー

**High（高）**：フェーズが計画通りに完了できない

- 必須質問への回答が不完全
- 矛盾したユーザー回答
- 前のフェーズからの依存関係が欠けている

**Medium（中）**：回避策でフェーズを継続できる

- オプションの成果物が欠けている
- 重大でない検証の失敗
- 部分的な完了が可能

**Low（低）**：進捗を阻止しないマイナーな問題

- フォーマットの不一致
- オプション情報が欠けている
- 非ブロッキングの警告

## フェーズ別エラー処理

### コンテキスト評価エラー

**エラー**：ワークスペースファイルを読み取れない

- **原因**：権限の問題、ディレクトリが存在しない
- **解決策**：ユーザーにワークスペースのパスと権限を確認するよう依頼する
- **回避策**：ユーザー提供の情報のみで進める

**エラー**：既存の`aidlc-state.md`が破損している

- **原因**：手動編集、以前の実行の不完全な終了
- **解決策**：ユーザーに最初からやり直すか回復を試みるかを確認する
- **回復**：バックアップを作成し、新しい状態ファイルを開始する

**エラー**：必要なフェーズを決定できない

- **原因**：ユーザーからの情報が不十分
- **解決策**：意図とスコープについて明確化の質問をする
- **回避策**：デフォルトとして包括的な実行計画にする

### 要件評価エラー

**エラー**：ユーザーが矛盾した要件を提供する

- **原因**：理解が不明確、ニーズが変化している
- **解決策**：矛盾を解決するためのフォローアップ質問を作成する
- **進めない**：矛盾が解決されるまで

**エラー**：要件ドキュメントを変換できない

- **原因**：サポートされていない形式、破損したファイル
- **解決策**：サポートされた形式で要件を提供するようユーザーに依頼する
- **回避策**：ユーザーの口頭での説明で作業する

**エラー**：検証質問への回答が不完全

- **原因**：ユーザーが質問をスキップした、何を答えるかが不明確
- **解決策**：未回答の質問を強調し、例を提供する
- **進めない**：すべての必須質問に回答されるまで

### ストーリー開発エラー

**エラー**：要件をストーリーにマップできない

- **原因**：要件が曖昧すぎる、機能的な詳細が欠けている
- **解決策**：明確化のために要件評価に戻る
- **回避策**：利用可能な情報に基づいてストーリーを作成し、不完全としてマーク

**エラー**：ユーザーが曖昧なストーリープランニングの回答を提供する

- **原因**：オプションが不明確、複雑な決定
- **解決策**：具体的な例を使ったフォローアップ質問を追加する
- **進めない**：曖昧さが解決されるまで

**エラー**：ストーリー生成計画に未完了のステップがある

- **原因**：実行が中断された、ステップがスキップされた
- **解決策**：最初の未完了ステップから再開する
- **回復**：完了したステップを確認し、チェックポイントから継続する

### アプリケーション設計エラー

**エラー**：アーキテクチャの決定が不明確または矛盾している

- **原因**：曖昧な回答、相反する要件
- **解決策**：決定を明確にするためのフォローアップ質問を追加する
- **進めない**：決定が明確になって文書化されるまで

**エラー**：サービス/ユニットの数を決定できない

- **原因**：境界についての情報が不十分
- **解決策**：デプロイ、チーム構造、スケーリングについて具体的な質問をする
- **回避策**：デフォルトとしてモノリスにし、後で変更できるようにする

### 設計エラー

**エラー**：ユニットの依存関係が循環している

- **原因**：境界定義が不良、密結合
- **解決策**：循環依存を特定し、リファクタリングを提案する
- **回復**：サイクルを断ち切るためにユニット境界を修正する

**エラー**：ユニット設計計画にステップが欠けている

- **原因**：計画生成が不完全、テンプレートエラー
- **解決策**：必要なすべてのステップで計画を再生成する
- **回復**：既存の計画に欠けているステップを追加する

**エラー**：設計成果物を生成できない

- **原因**：ユニット情報が欠けている、要件が不明確
- **解決策**：ユニット定義を明確にするためにユニットプランニングに戻る
- **回避策**：部分的な設計を生成し、ギャップをマーク

### NFR実装エラー

**エラー**：テクノロジースタックの選択が互換性がない

- **原因**：相反する要件、プラットフォームの制限
- **解決策**：非互換性を強調し、ユーザーに選択を求める
- **進めない**：互換性のある選択が行われるまで

**エラー**：組織の制約を満たせない

- **原因**：ネットワーク制限、セキュリティポリシー
- **解決策**：制約を文書化し、回避策をユーザーに求める
- **エスカレーション**：セットアップのために人間の介入が必要な場合がある

**エラー**：NFR実装ステップに人間のアクションが必要

- **原因**：AIが特定のタスクを実行できない（ネットワーク設定、認証情報）
- **解決策**：**人間のタスク（HUMAN TASK）**として明確にマークし、手順を提供する
- **待機**：先に進む前にユーザーの確認を待つ

### コードプランニングエラー

**エラー**：コード生成計画が不完全

- **原因**：設計成果物が欠けている、要件が不明確
- **解決策**：成果物を完了するために設計フェーズに戻る
- **回復**：利用可能な情報で計画を生成し、ギャップをマーク

**エラー**：ユニットの依存関係が満たされていない

- **原因**：依存するユニットがまだ生成されていない
- **解決策**：依存関係を考慮して生成シーケンスを並べ直す
- **回避策**：スタブ依存関係で生成し、後で統合する

### コード生成エラー

**エラー**：あるステップのコードを生成できない

- **原因**：設計情報が不十分、要件が不明確
- **解決策**：ステップをスキップし、不完全としてマークし、継続する
- **回復**：より多くの情報を収集した後にステップに戻る

**エラー**：生成されたコードに構文エラーがある

- **原因**：テンプレートの問題、言語固有の問題
- **解決策**：構文エラーを修正し、必要に応じて再生成する
- **検証**：先に進む前にコードがコンパイルされることを確認する

**エラー**：テスト生成が失敗する

- **原因**：複雑なロジック、テストフレームワークの設定が欠けている
- **解決策**：基本的なテスト構造を生成し、手動完了としてマーク
- **回避策**：テストなしで進め、オペレーションフェーズで追加する

### オペレーションエラー

**エラー**：ビルドツールを特定できない

- **原因**：通常とは異なるプロジェクト構造、複数のビルドシステム
- **解決策**：ビルドツールとコマンドを指定するようユーザーに依頼する
- **回避策**：汎用的な手順を提供し、ユーザーが適応する

**エラー**：デプロイターゲットが不明

- **原因**：複数の環境、複雑なインフラストラクチャ
- **解決策**：デプロイターゲットとメソッドを指定するようユーザーに依頼する
- **回避策**：一般的なプラットフォームの手順を提供する

## 回復手順

### 部分的なフェーズ完了

**シナリオ**：フェーズが実行の途中で中断された

**回復ステップ**：

1. フェーズ計画ファイルを読み込む
2. 最後に完了したステップを特定する（最後の[x]チェックボックス）
3. 次の未完了ステップから再開する
4. 以前のすべてのステップが実際に完了していることを確認する
5. 実行を通常通り継続する

### 破損した状態ファイル

**シナリオ**：`aidlc-state.md`が破損または一貫性がない

**回復ステップ**：

1. バックアップを作成：`aidlc-state.md.backup`
2. ユーザーに実際にどのフェーズにいるかを確認する
3. 状態ファイルを最初から再生成する
4. 既存の成果物に基づいて完了したフェーズをマークする
5. 現在のフェーズから再開する

### 欠けている成果物

**シナリオ**：前のフェーズから必要な成果物が欠けている

**回復ステップ**：

1. どの成果物が欠けているかを特定する
2. 再生成できるかどうかを判断する
3. 可能な場合：そのフェーズに戻り、成果物を再生成する
4. 不可能な場合：ユーザーに手動で情報を提供するよう依頼する
5. ギャップを`audit.md`に文書化する

### ユーザーがフェーズを再開したい場合

**シナリオ**：ユーザーがフェーズの結果に不満でやり直したい

**回復ステップ**：

1. ユーザーが再開したいことを確認する（データが失われる）
2. 既存の成果物をアーカイブ：`{成果物}.backup`
3. `aidlc-state.md`のフェーズステータスをリセット
4. 計画ファイルのフェーズチェックボックスをクリア
5. 最初からフェーズを再実行する

### ユーザーがフェーズをスキップしたい場合

**シナリオ**：ユーザーが計画されていたフェーズをスキップしたい

**回復ステップ**：

1. ユーザーが影響を理解していることを確認する
2. スキップの理由を`audit.md`に文書化する
3. `aidlc-state.md`でフェーズを「スキップ済み（SKIPPED）」としてマーク
4. 次のフェーズに進む
5. 注意：依存関係が欠けている場合、後のフェーズで問題が発生する可能性がある

## エスカレーションガイドライン

### ユーザーの助けを求める場合

**即座に**：

- 矛盾したまたは曖昧なユーザー入力
- 必要な情報が欠けている
- AIが解決できない技術的制約
- ビジネス判断が必要な決定

**解決を試みた後**：

- 同じステップで繰り返しエラー
- 複雑な技術的問題
- 通常とは異なるプロジェクト構造
- 外部システムとの統合

### 最初からやり直しを提案する場合

**以下の場合は再スタートを検討する**：

- 複数のフェーズでエラーが発生している
- 状態ファイルが深刻に破損している
- ユーザー要件が大幅に変更された
- アーキテクチャの決定を覆す必要がある
- ユーザーが欠けている情報を提供できない
- フェーズ全体で成果物が一貫していない

**やり直しの前に**：

1. 既存の作業をすべてアーカイブする
2. 学んだ教訓を文書化する
3. 何を保存するかを特定する
4. ユーザーの確認を得る
5. 新しい実行計画を作成する

## セッション再開エラー

### 再開中の欠けている成果物

**エラー**：以前のステージから必要な成果物が欠けている

- **原因**：ファイルが削除された、移動された、または作成されなかった
- **解決策**：
  1. どのステージが欠けている成果物を作成したかを特定する
  2. aidlc-state.mdでステージが完了とマークされているか確認する
  3. 完了とマークされているが成果物が欠けている場合：そのステージを再生成する
  4. 完了とマークされていない場合：そのステージから再開する
- **回復**：欠けている成果物を作成するステージに戻って再実行する

**エラー**：成果物ファイルが存在するが空または破損している

- **原因**：書き込みの中断、手動編集、ファイルシステムの問題
- **解決策**：
  1. 破損したファイルのバックアップを作成する
  2. それを作成するステージから再生成を試みる
  3. 再生成できない場合：再作成するための情報をユーザーに求める
- **回復**：成果物を作成するステージを再実行する

### 再開中の一貫性のない状態

**エラー**：aidlc-state.mdがステージ完了を示しているが成果物が存在しない

- **原因**：状態ファイルが更新されたが成果物の生成が失敗した
- **解決策**：
  1. aidlc-state.mdでステージを未完了としてマークする
  2. 成果物を生成するためにステージを再実行する
  3. 完了とマークする前に成果物が存在することを確認する
- **回復**：ステージのステータスをリセットして再実行する

**エラー**：成果物が存在するがaidlc-state.mdがステージを未完了として示している

- **原因**：成果物の生成は成功したが状態の更新が失敗した
- **解決策**：
  1. 成果物が完全で有効であることを確認する
  2. aidlc-state.mdを更新してステージを完了としてマークする
  3. 次のステージに進む
- **回復**：実際の完了を反映するために状態ファイルを更新する

**エラー**：aidlc-state.mdで複数のステージが「現在（current）」としてマークされている

- **原因**：状態ファイルの破損、手動編集
- **解決策**：
  1. 成果物を確認して実際の進捗を判断する
  2. 実際にどのステージにいるかをユーザーに確認する
  3. aidlc-state.mdを修正して単一の現在のステージを示す
- **回復**：既存の成果物に基づいて状態ファイルを再構築する

### コンテキスト読み込みエラー

**エラー**：以前のステージから必要なコンテキストを読み込めない

- **原因**：ファイルが欠けている、コンテンツが破損している、ファイルパスが間違っている
- **解決策**：
  1. 現在のステージに必要な成果物をリストアップする
  2. 欠けているまたは破損しているものを確認する
  3. 欠けている成果物を再生成するかユーザーに情報を求める
- **回復**：現在のステージを再開する前に前提条件のステージを完了する

**エラー**：読み込まれた成果物に矛盾した情報が含まれている

- **原因**：手動編集、複数の人が作業している、不完全な更新
- **解決策**：
  1. 矛盾を特定してユーザーに提示する
  2. どの情報が正しいかをユーザーに確認する
  3. 矛盾を解決するために成果物を更新する
- **回復**：先に進む前に矛盾を調整する

### 再開のベストプラクティス

1. **常に状態を検証**：aidlc-state.mdが実際の成果物と一致しているか確認する
2. **段階的に読み込む**：ステージごとに成果物を読み込み、それぞれを検証する
3. **早期に失敗させる**：重要な成果物が欠けている場合は即座に停止する
4. **明確にコミュニケーション**：何が欠けていてなぜ必要かをユーザーに正確に伝える
5. **オプションを提示する**：再生成、手動提供、または最初からやり直しを提案する
6. **回復を文書化**：すべての回復アクションをaudit.mdにログ記録する

## ログ記録要件

### エラーログのフォーマット

```markdown
## エラー - [フェーズ名]
**タイムスタンプ**：[ISOタイムスタンプ]
**エラータイプ**：[Critical/High/Medium/Low]
**説明**：[何が問題になったか]
**原因**：[なぜ発生したか]
**解決策**：[どのように解決されたか]
**影響**：[ワークフローへの影響]

---
```

### 回復ログのフォーマット

```markdown
## 回復 - [フェーズ名]
**タイムスタンプ**：[ISOタイムスタンプ]
**問題**：[何の回復が必要だったか]
**回復ステップ**：[何をしたか]
**結果**：[回復の結果]
**影響を受けた成果物**：[ファイルのリスト]

---
```

## 予防のベストプラクティス

1. **早期に検証する**：作業を始める前に入力と依存関係を確認する
2. **頻繁にチェックポイントを設ける**：ステップを完了した直後にチェックボックスを更新する
3. **明確にコミュニケーションする**：何をしているか、なぜしているかを説明する
4. **質問する**：推測しない——曖昧さはすぐに明確にする
5. **すべてを文書化する**：すべての決定と変更を`audit.md`にログ記録する
