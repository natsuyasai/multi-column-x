# NFR要件

## 前提条件

- ユニットの機能設計が完了していること
- ユニットの機能設計成果物が利用可能であること
- 実行計画がNFR要件ステージを実行すべきことを示していること

## 概要

ユニットの非機能要件を決定し、テクスタックの選択を行う。

## 実行ステップ

### ステップ1：機能設計の分析

- `aidlc-docs/construction/{ユニット名}/functional-design/`から機能設計成果物を読み込む
- ビジネスロジックの複雑さと要件を理解する

### ステップ2：NFR要件計画の作成

- NFR評価のためのチェックボックス[]付きの計画を生成する
- スケーラビリティ、パフォーマンス、可用性、セキュリティに焦点を当てる
- 各ステップにはチェックボックス[]を含める

### ステップ3：コンテキストに適した質問の生成

**指示**：機能設計を徹底的に分析して、システム品質とアーキテクチャの決定を改善するためにNFRの明確化が必要なすべての領域を特定する。システム品質を確保するために積極的に質問を行う。

**重要**：システム品質に影響する可能性のある曖昧さや欠けている詳細がある場合は、デフォルトで質問する。間違ったNFRの仮定をするよりも多くの質問をする方が良い。

- [Answer]:タグ形式を使用して質問を埋め込む
- 曖昧さや欠けている情報、明確化が必要な領域に焦点を当てる
- NFRとテクスタックの決定を改善するユーザー入力が必要な箇所に質問を生成する
- **迷ったら質問する** - 過信はシステム品質の低下につながる

**評価すべき質問カテゴリー**（すべてのカテゴリーを考慮する）：

- **スケーラビリティ要件** - 期待される負荷、成長パターン、スケーリングのトリガー、キャパシティプランニングについて質問する
- **パフォーマンス要件** - 応答時間、スループット、レイテンシー、パフォーマンスベンチマークについて質問する
- **可用性要件** - アップタイムの期待値、ディザスタリカバリー、フェイルオーバー、ビジネス継続性について質問する
- **セキュリティ要件** - データ保護、コンプライアンス、認証、認可、脅威モデルについて質問する
- **テクスタックの選択** - 技術的な好み、制約、既存システム、統合要件について質問する
- **信頼性要件** - エラー処理、耐障害性、モニタリング、アラートのニーズについて質問する
- **保守性要件** - コード品質、ドキュメント、テスト、運用要件について質問する
- **使いやすさ要件** - ユーザーエクスペリエンス、アクセシビリティ、インターフェース要件について質問する

### ステップ4：計画の保存

- `aidlc-docs/construction/plans/{ユニット名}-nfr-requirements-plan.md`として保存する
- ユーザー入力のためのすべての[Answer]:タグを含める

### ステップ5：回答の収集と分析

- ユーザーがすべての[Answer]:タグを完了するのを待つ
- **必須**：曖昧または不明確な回答についてすべての回答を注意深く確認する
- **重要**：不明確な回答に対してフォローアップ質問を追加する——曖昧さのまま進まない
- 「depends」、「maybe」、「not sure」、「mix of」、「somewhere between」、「standard」、「typical」などの回答を探す
- 曖昧さが検出された場合は明確化質問ファイルを作成する
- **すべての曖昧さが解決されるまで進まない**

### ステップ6：NFR要件成果物の生成

- `aidlc-docs/construction/{ユニット名}/nfr-requirements/nfr-requirements.md`を作成する
- `aidlc-docs/construction/{ユニット名}/nfr-requirements/tech-stack-decisions.md`を作成する

### ステップ7：完了メッセージの提示

- 以下の構造で完了メッセージを提示する：
     1. **完了アナウンス**（必須）：常にこれで始める：

```markdown
# 📊 NFR要件完了 - [ユニット名]
```

     2. **AIサマリー**（任意）：NFR要件の構造化された箇条書きサマリーを提供する
        - フォーマット：「NFR要件評価で[説明]が特定されました：」
        - 主要なスケーラビリティ、パフォーマンス、可用性要件をリスト（箇条書き）
        - 特定されたセキュリティとコンプライアンス要件をリスト
        - テクスタックの決定と根拠を言及する
        - ワークフロー指示を含めない（「レビューしてください」、「お知らせください」、「次のフェーズに進む」、「進む前に」）
        - 事実に基づいてコンテンツに焦点を当てる
     3. **フォーマットされたワークフローメッセージ**（必須）：常にこの正確なフォーマットで終わる：

```markdown
> **📋 <u>**レビュー必須：**</u>**
> NFR要件を確認してください：`aidlc-docs/construction/[ユニット名]/nfr-requirements/`



> **🚀 <u>**次は何をしますか？**</u>**
>
> **選択してください：**
>
> 🔧 **変更のリクエスト** - レビューに基づいてNFR要件の修正をリクエストする
> ✅ **次のステージに進む** - NFR要件を承認して**[次のステージ名]**に進む

---
```

### ステップ8：明示的な承認の待機

- ユーザーがNFR要件を明示的に承認するまで進まない
- 承認は明確で曖昧さがないこと
- ユーザーが変更をリクエストした場合、要件を更新して承認プロセスを繰り返す

### ステップ9：承認の記録と進捗の更新

- タイムスタンプ付きでaudit.mdに承認をログ記録する
- ユーザーの承認回答をタイムスタンプ付きで記録する
- aidlc-state.mdでNFR要件ステージを完了としてマークする
