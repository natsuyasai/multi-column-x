# 機能設計

## 目的

**ユニットごとの詳細なビジネスロジック設計**

機能設計の焦点：

- ユニットの詳細なビジネスロジックとアルゴリズム
- エンティティと関係を持つドメインモデル
- 詳細なビジネスルール、バリデーションロジック、制約
- テクノロジー非依存の設計（インフラストラクチャの懸念なし）

**注**：これはアプリケーション設計（INCEPTIONフェーズ）の高レベルのコンポーネント設計を基盤として構築される

## 前提条件

- ユニット生成が完了していること
- 作業単位の成果物が利用可能であること
- アプリケーション設計を推奨（高レベルのコンポーネント構造を提供する）
- 実行計画が機能設計ステージを実行すべきことを示していること

## 概要

ユニットの詳細なビジネスロジックを設計する。テクノロジー非依存でビジネス機能に純粋に焦点を当てる。

## 実行ステップ

### ステップ1：ユニットコンテキストの分析

- `aidlc-docs/inception/application-design/unit-of-work.md`からユニット定義を読み込む
- `aidlc-docs/inception/application-design/unit-of-work-story-map.md`から割り当てられたストーリーを読み込む
- ユニットの責任と境界を理解する

### ステップ2：機能設計計画の作成

- 機能設計のためのチェックボックス[]付きの計画を生成する
- ビジネスロジック、ドメインモデル、ビジネスルールに焦点を当てる
- 各ステップにはチェックボックス[]を含める

### ステップ3：コンテキストに適した質問の生成

**指示**：ユニット定義と機能設計成果物を徹底的に分析して、機能設計の品質を改善する明確化が役立つすべての領域を特定する。包括的な理解を確保するために積極的に質問を行う。

**重要**：機能設計の品質に影響する可能性のある曖昧さや欠けている詳細がある場合は、デフォルトで質問する。設計の悪い結果につながるよりも多くの質問をする方が良い。

- [Answer]:タグ形式を使用して質問を埋め込む
- 曖昧さ、欠けている情報、または明確化が必要な領域に焦点を当てる
- 機能設計の決定を改善するユーザー入力が役立つ箇所に質問を生成する
- **迷ったら質問する** - 過信は設計の悪化につながる

**評価すべき質問カテゴリー**（すべてのカテゴリーを評価する）：

- **ビジネスロジックモデリング** - コアエンティティ、ワークフロー、データ変換、ビジネスプロセスについて質問する
- **ドメインモデル** - ドメインの概念、エンティティの関係、データ構造、ビジネスオブジェクトについて質問する
- **ビジネスルール** - 決定ルール、バリデーションロジック、制約、ビジネスポリシーについて質問する
- **データフロー** - データの入力、出力、変換、永続化要件について質問する
- **統合ポイント** - 外部システムとのインタラクション、API、データ交換について質問する
- **エラー処理** - エラーシナリオ、バリデーション失敗、例外処理について質問する
- **ビジネスシナリオ** - エッジケース、代替フロー、複雑なビジネス状況について質問する
- **フロントエンドコンポーネント**（該当する場合）- UIコンポーネントの構造、ユーザーインタラクション、状態管理、フォーム処理について質問する

### ステップ4：計画の保存

- `aidlc-docs/construction/plans/{ユニット名}-functional-design-plan.md`として保存する
- ユーザー入力のためのすべての[Answer]:タグを含める

### ステップ5：回答の収集と分析

- ユーザーがすべての[Answer]:タグを完了するのを待つ
- **必須**：曖昧または不明確な回答についてすべての回答を注意深く確認する
- **重要**：不明確な回答に対してフォローアップ質問を追加する——曖昧さのまま進まない
- 「depends」、「maybe」、「not sure」、「mix of」、「somewhere between」などの回答を探す
- 曖昧さが検出された場合は明確化質問ファイルを作成する
- **すべての曖昧さが解決されるまで進まない**

### ステップ6：機能設計成果物の生成

- `aidlc-docs/construction/{ユニット名}/functional-design/business-logic-model.md`を作成する
- `aidlc-docs/construction/{ユニット名}/functional-design/business-rules.md`を作成する
- `aidlc-docs/construction/{ユニット名}/functional-design/domain-entities.md`を作成する
- ユニットにフロントエンド/UIが含まれる場合：`aidlc-docs/construction/{ユニット名}/functional-design/frontend-components.md`を作成する
  - コンポーネントの階層と構造
  - 各コンポーネントのPropsと状態定義
  - ユーザーインタラクションフロー
  - フォームバリデーションルール
  - API統合ポイント（各コンポーネントが使用するバックエンドエンドポイント）

### ステップ7：完了メッセージの提示

- 以下の構造で完了メッセージを提示する：
     1. **完了アナウンス**（必須）：常にこれで始める：

```markdown
# 🔧 機能設計完了 - [ユニット名]
```

     2. **AIサマリー**（任意）：機能設計の構造化された箇条書きサマリーを提供する
        - フォーマット：「機能設計で[説明]が作成されました：」
        - 主要なビジネスロジックモデルとエンティティをリスト（箇条書き）
        - 定義されたビジネスルールとバリデーションロジックをリスト
        - ドメインモデルの構造と関係を言及する
        - ワークフロー指示を含めない（「レビューしてください」、「お知らせください」、「次のフェーズに進む」、「進む前に」）
        - 事実に基づいてコンテンツに焦点を当てる
     3. **フォーマットされたワークフローメッセージ**（必須）：常にこの正確なフォーマットで終わる：

```markdown
> **📋 <u>**レビュー必須：**</u>**
> 機能設計成果物を確認してください：`aidlc-docs/construction/[ユニット名]/functional-design/`



> **🚀 <u>**次は何をしますか？**</u>**
>
> **選択してください：**
>
> 🔧 **変更のリクエスト** - レビューに基づいて機能設計の修正をリクエストする
> ✅ **次のステージに進む** - 機能設計を承認して**[次のステージ名]**に進む

---
```

### ステップ8：明示的な承認の待機

- ユーザーが機能設計を明示的に承認するまで進まない
- 承認は明確で曖昧さがないこと
- ユーザーが変更をリクエストした場合、設計を更新して承認プロセスを繰り返す

### ステップ9：承認の記録と進捗の更新

- タイムスタンプ付きでaudit.mdに承認をログ記録する
- ユーザーの承認回答をタイムスタンプ付きで記録する
- aidlc-state.mdで機能設計ステージを完了としてマークする
