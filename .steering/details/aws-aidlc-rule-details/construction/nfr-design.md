# NFR設計

## 前提条件

- ユニットのNFR要件が完了していること
- NFR要件成果物が利用可能であること
- 実行計画がNFR設計ステージを実行すべきことを示していること

## 概要

パターンと論理コンポーネントを使用してNFR要件をユニット設計に組み込む。

## 実行ステップ

### ステップ1：NFR要件の分析

- `aidlc-docs/construction/{ユニット名}/nfr-requirements/`からNFR要件を読み込む
- スケーラビリティ、パフォーマンス、可用性、セキュリティのニーズを理解する

### ステップ2：NFR設計計画の作成

- NFR設計のためのチェックボックス[]付きの計画を生成する
- 設計パターンと論理コンポーネントに焦点を当てる
- 各ステップにはチェックボックス[]を含める

### ステップ3：コンテキストに適した質問の生成

**指示**：NFR要件を分析して、このユニットのNFR設計に関連する質問のみを生成する。以下のカテゴリーを必須チェックリストではなくインスピレーションとして使用する。該当しないカテゴリーはスキップする。

- [Answer]:タグ形式を使用して質問を埋め込む
- このユニット固有の曖昧さや欠けている情報に焦点を当てる
- パターンとコンポーネントの決定にユーザー入力が必要な箇所にのみ質問を生成する

**質問カテゴリーの例**（必要に応じて適応する）：

- **レジリエンスパターン** - 耐障害性アプローチの明確化が必要な場合のみ
- **スケーラビリティパターン** - スケーリングメカニズムが不明確な場合のみ
- **パフォーマンスパターン** - パフォーマンス最適化戦略が曖昧な場合のみ
- **セキュリティパターン** - セキュリティ実装アプローチにユーザー入力が必要な場合のみ
- **論理コンポーネント** - インフラストラクチャコンポーネント（キュー、キャッシュなど）の明確化が必要な場合のみ

### ステップ4：計画の保存

- `aidlc-docs/construction/plans/{ユニット名}-nfr-design-plan.md`として保存する
- ユーザー入力のためのすべての[Answer]:タグを含める

### ステップ5：回答の収集と分析

- ユーザーがすべての[Answer]:タグを完了するのを待つ
- 曖昧または不明確な回答を確認する
- 必要に応じてフォローアップ質問を追加する

### ステップ6：NFR設計成果物の生成

- `aidlc-docs/construction/{ユニット名}/nfr-design/nfr-design-patterns.md`を作成する
- `aidlc-docs/construction/{ユニット名}/nfr-design/logical-components.md`を作成する

### ステップ7：完了メッセージの提示

- 以下の構造で完了メッセージを提示する：
     1. **完了アナウンス**（必須）：常にこれで始める：

```markdown
# 🎨 NFR設計完了 - [ユニット名]
```

     2. **AIサマリー**（任意）：NFR設計の構造化された箇条書きサマリーを提供する
        - フォーマット：「NFR設計で[説明]が組み込まれました：」
        - 実装された主要な設計パターンをリスト（箇条書き）
        - 論理コンポーネントとインフラストラクチャ要素をリスト
        - 適用されたレジリエンス、スケーラビリティ、パフォーマンスパターンを言及する
        - ワークフロー指示を含めない（「レビューしてください」、「お知らせください」、「次のフェーズに進む」、「進む前に」）
        - 事実に基づいてコンテンツに焦点を当てる
     3. **フォーマットされたワークフローメッセージ**（必須）：常にこの正確なフォーマットで終わる：

```markdown
> **📋 <u>**レビュー必須：**</u>**
> NFR設計を確認してください：`aidlc-docs/construction/[ユニット名]/nfr-design/`



> **🚀 <u>**次は何をしますか？**</u>**
>
> **選択してください：**
>
> 🔧 **変更のリクエスト** - レビューに基づいてNFR設計の修正をリクエストする
> ✅ **次のステージに進む** - NFR設計を承認して**[次のステージ名]**に進む

---
```

### ステップ8：明示的な承認の待機

- ユーザーがNFR設計を明示的に承認するまで進まない
- 承認は明確で曖昧さがないこと
- ユーザーが変更をリクエストした場合、設計を更新して承認プロセスを繰り返す

### ステップ9：承認の記録と進捗の更新

- タイムスタンプ付きでaudit.mdに承認をログ記録する
- ユーザーの承認回答をタイムスタンプ付きで記録する
- aidlc-state.mdでNFR設計ステージを完了としてマークする
