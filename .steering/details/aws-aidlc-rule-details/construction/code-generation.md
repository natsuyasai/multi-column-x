# コード生成 - 詳細ステップ

## 概要

このステージでは2つの統合されたパートを通じて各作業単位のコードを生成する：

- **パート1 - プランニング**：明示的なステップを含む詳細なコード生成計画の作成
- **パート2 - 生成**：承認された計画を実行してコード、テスト、成果物を生成する

**注**：ブラウンフィールドプロジェクトでは「生成」とは重複ファイルを作成するのではなく、必要に応じて既存のファイルを修正することを意味する。

## 前提条件

- ユニットのユニット設計生成が完了していること
- NFR実装（実行された場合）がユニットに対して完了していること
- すべてのユニット設計成果物が利用可能であること
- ユニットがコード生成の準備ができていること

---

# パート1：プランニング

## ステップ1：ユニットコンテキストの分析

- [ ] ユニット設計生成からユニット設計成果物を読み込む
- [ ] 割り当てられたストーリーを理解するためにユニットのストーリーマップを読み込む
- [ ] ユニットの依存関係とインターフェースを特定する
- [ ] ユニットがコード生成の準備ができているかを検証する

## ステップ2：詳細なユニットコード生成計画の作成

- [ ] `aidlc-docs/aidlc-state.md`からワークスペースのルートとプロジェクトタイプを読み込む
- [ ] コードの場所を決定する（構造パターンについては重要ルールを参照）
- [ ] **ブラウンフィールドのみ**：リバースエンジニアリングのcode-structure.mdで変更する既存ファイルを確認する
- [ ] 正確なパスを文書化する（aidlc-docs/以外）
- [ ] ユニット生成の明示的なステップを作成する：
  - プロジェクト構造のセットアップ（グリーンフィールドのみ）
  - ビジネスロジックの生成
  - ビジネスロジックのユニットテスト
  - ビジネスロジックのサマリー
  - APIレイヤーの生成
  - APIレイヤーのユニットテスト
  - APIレイヤーのサマリー
  - リポジトリレイヤーの生成
  - リポジトリレイヤーのユニットテスト
  - リポジトリレイヤーのサマリー
  - フロントエンドコンポーネントの生成（該当する場合）
  - フロントエンドコンポーネントのユニットテスト（該当する場合）
  - フロントエンドコンポーネントのサマリー（該当する場合）
  - データベースマイグレーションスクリプト（データモデルが存在する場合）
  - ドキュメントの生成（APIドキュメント、READMEの更新）
  - デプロイ成果物の生成
- [ ] 各ステップに順番に番号を付ける
- [ ] ストーリーマッピング参照を含める
- [ ] 各ステップにチェックボックス[]を追加する

## ステップ3：ユニット生成コンテキストの含有

- [ ] このユニットについて以下を含める：
  - このユニットが実装するストーリー
  - 他のユニット/サービスへの依存関係
  - 期待されるインターフェースとコントラクト
  - このユニットが所有するデータベースエンティティ
  - サービスの境界と責任

## ステップ4：ユニット計画ドキュメントの作成

- [ ] 完全な計画を`aidlc-docs/construction/plans/{ユニット名}-code-generation-plan.md`として保存する
- [ ] ステップ番号付けを含める（ステップ1、ステップ2など）
- [ ] ユニットコンテキストと依存関係を含める
- [ ] ストーリーのトレーサビリティを含める
- [ ] 計画がステップバイステップで実行可能であることを確認する
- [ ] この計画がコード生成の唯一の信頼できるソースであることを強調する

## ステップ5：ユニット計画のサマリー

- [ ] ユニットコード生成計画のサマリーをユーザーに提供する
- [ ] ユニット生成アプローチを強調する
- [ ] ステップシーケンスとストーリーカバレッジを説明する
- [ ] ステップの総数と推定スコープを注記する

## ステップ6：承認プロンプトのログ記録

- [ ] 承認を求める前に、タイムスタンプ付きでプロンプトを`aidlc-docs/audit.md`にログ記録する
- [ ] 完全なユニットコード生成計画への参照を含める
- [ ] ISO 8601タイムスタンプ形式を使用する

## ステップ7：明示的な承認の待機

- [ ] ユーザーがユニットコード生成計画を明示的に承認するまで進まない
- [ ] 承認は計画と生成シーケンス全体をカバーする必要がある
- [ ] ユーザーが変更をリクエストした場合、計画を更新して承認プロセスを繰り返す

## ステップ8：承認回答の記録

- [ ] タイムスタンプ付きでユーザーの承認回答を`aidlc-docs/audit.md`にログ記録する
- [ ] 正確なユーザー回答テキストを含める
- [ ] 承認ステータスを明確にマークする

## ステップ9：進捗の更新

- [ ] aidlc-state.mdでコードプランニングを完了としてマーク
- [ ] 「現在のステータス」セクションを更新する
- [ ] コード生成への移行を準備する

# パート2：生成

## ステップ10：ユニットコード生成計画の読み込み

- [ ] `aidlc-docs/construction/plans/{ユニット名}-code-generation-plan.md`から完全な計画を読み込む
- [ ] 次の未完了ステップを特定する（最初の[ ]チェックボックス）
- [ ] そのステップのコンテキスト（ユニット、依存関係、ストーリー）を読み込む

## ステップ11：現在のステップの実行

- [ ] 計画からターゲットディレクトリを確認する（aidlc-docs/以外）
- [ ] **ブラウンフィールドのみ**：ターゲットファイルが存在するか確認する
- [ ] 現在のステップが説明する内容を正確に生成する：
  - **ファイルが存在する場合**：インプレースで修正する（`ClassName_modified.java`、`ClassName_new.java`などを作成しない）
  - **ファイルが存在しない場合**：新しいファイルを作成する
- [ ] 正しい場所に書き込む：
  - **アプリケーションコード**：プロジェクト構造に従ってワークスペースのルート
  - **ドキュメント**：`aidlc-docs/construction/{ユニット名}/code/`（Markdownのみ）
  - **ビルド/設定ファイル**：ワークスペースのルート
- [ ] ユニットのストーリー要件に従う
- [ ] 依存関係とインターフェースを尊重する

## ステップ12：進捗の更新

- [ ] ユニットコード生成計画で完了したステップを[x]としてマークする
- [ ] 機能の実装が完了したらユニットのストーリーを[x]としてマークする
- [ ] `aidlc-docs/aidlc-state.md`の現在のステータスを更新する
- [ ] **ブラウンフィールドのみ**：重複ファイルが作成されていないか確認する（例：`ClassName.java`と並んで`ClassName_modified.java`がないか）
- [ ] 生成されたすべての成果物を保存する

## ステップ13：生成の継続または完了

- [ ] 残りのステップがある場合はステップ10に戻る
- [ ] すべてのステップが完了した場合は完了メッセージの提示に進む

## ステップ14：完了メッセージの提示

- 以下の構造で完了メッセージを提示する：
     1. **完了アナウンス**（必須）：常にこれで始める：

```markdown
# 💻 コード生成完了 - [ユニット名]
```

     2. **AIサマリー**（任意）：構造化された箇条書きサマリーを提供する
        - **ブラウンフィールド**：修正したファイルと作成したファイルを区別する（例：「• 修正：`src/services/user-service.ts`」、「• 作成：`src/services/auth-service.ts`」）
        - **グリーンフィールド**：作成したファイルをパス付きでリスト（例：「• 作成：`src/services/user-service.ts`」）
        - テスト、ドキュメント、デプロイ成果物をパス付きでリスト
        - 事実に基づいて、ワークフロー指示は含めない
     3. **フォーマットされたワークフローメッセージ**（必須）：常にこの正確なフォーマットで終わる：

```markdown
> **📋 <u>**レビュー必須：**</u>**
> 生成されたコードを確認してください：
> - **アプリケーションコード**：`[実際のワークスペースパス]`
> - **ドキュメント**：`aidlc-docs/construction/[ユニット名]/code/`



> **🚀 <u>**次は何をしますか？**</u>**
>
> **選択してください：**
>
> 🔧 **変更のリクエスト** - レビューに基づいて生成されたコードの修正をリクエストする
> ✅ **次のステージに進む** - コード生成を承認して**[次のユニット/ビルド＆テスト]**に進む

---
```

## ステップ15：明示的な承認の待機

- 生成されたコードをユーザーが明示的に承認するまで進まない
- 承認は明確で曖昧さがないこと
- ユーザーが変更をリクエストした場合、コードを更新して承認プロセスを繰り返す

## ステップ16：承認の記録と進捗の更新

- タイムスタンプ付きでaudit.mdに承認をログ記録する
- ユーザーの承認回答をタイムスタンプ付きで記録する
- aidlc-state.mdでこのユニットのコード生成ステージを完了としてマークする

## 重要ルール

### コードの場所ルール

- **アプリケーションコード**：ワークスペースのルートのみ（aidlc-docs/には絶対に置かない）
- **ドキュメント**：aidlc-docs/のみ（Markdownのサマリー）
- **ワークスペースのルートを読み込む**：コード生成前にaidlc-state.mdから

**プロジェクトタイプ別の構造パターン**：

- **ブラウンフィールド**：既存の構造を使用（例：`src/main/java/`、`lib/`、`pkg/`）
- **グリーンフィールド単一ユニット**：ワークスペースのルートに`src/`、`tests/`、`config/`
- **グリーンフィールドマルチユニット（マイクロサービス）**：`{ユニット名}/src/`、`{ユニット名}/tests/`
- **グリーンフィールドマルチユニット（モノリス）**：`src/{ユニット名}/`、`tests/{ユニット名}/`

### ブラウンフィールドのファイル修正ルール

- 生成前にファイルが存在するか確認する
- 存在する場合：インプレースで修正する（`ClassName_modified.java`などのコピーを作成しない）
- 存在しない場合：新しいファイルを作成する
- 生成後に重複ファイルがないか確認する（ステップ12）

### プランニングフェーズのルール

- すべての生成活動の明示的な番号付きステップを作成する
- 計画にストーリーのトレーサビリティを含める
- ユニットコンテキストと依存関係を文書化する
- 生成前にユーザーの明示的な承認を得る

### 生成フェーズのルール

- **ハードコードしたロジックは使用しない**：ユニット計画に記述されたことのみを実行する
- **計画に厳密に従う**：ステップシーケンスから逸脱しない
- **チェックボックスを更新する**：各ステップ完了直後に[x]をマークする
- **ストーリーのトレーサビリティ**：機能が実装されたらユニットのストーリーに[x]をマークする
- **依存関係を尊重する**：ユニットの依存関係が満たされた場合のみ実装する

### 自動化に適したコードルール

UIコード（ウェブ、モバイル、デスクトップ）を生成する際は、要素が自動化に適していることを確認する：

- インタラクティブな要素（ボタン、入力、リンク、フォーム）に`data-testid`属性を追加する
- 一貫した命名を使用：`{コンポーネント}-{要素の役割}`（例：`login-form-submit-button`、`user-list-search-input`）
- レンダリング間で変化する動的または自動生成のIDは避ける
- 要素の目的が変わった場合のみ変更するように`data-testid`値を安定させる

## 完了基準

- 完全なユニットコード生成計画が作成され承認されている
- ユニットコード生成計画のすべてのステップが[x]でマークされている
- 計画に従って、すべてのユニットのストーリーが実装されている
- すべてのコードとテストが生成されている（テストはビルド＆テストフェーズで実行される）
- デプロイ成果物が生成されている
- 完全なユニットがビルドと検証の準備ができている
