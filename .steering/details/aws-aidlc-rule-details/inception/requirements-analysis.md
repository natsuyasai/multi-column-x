# 要件分析（適応型）

**役割**：プロダクトオーナーとして行動する

**適応フェーズ**：常に実行される。詳細レベルは問題の複雑さに応じて適応する。

**適応深度の説明については[depth-levels.md](../common/depth-levels.md)を参照してください**

## 前提条件

- ワークスペース検出が完了していること
- リバースエンジニアリングが完了していること（ブラウンフィールドの場合）

## 実行ステップ

### ステップ1：リバースエンジニアリングのコンテキストの読み込み（利用可能な場合）

**ブラウンフィールドプロジェクトの場合**：

- `aidlc-docs/inception/reverse-engineering/architecture.md`を読み込む
- `aidlc-docs/inception/reverse-engineering/component-inventory.md`を読み込む
- `aidlc-docs/inception/reverse-engineering/technology-stack.md`を読み込む
- リクエストを分析する際に既存システムを理解するためにこれらを使用する

### ステップ2：ユーザーリクエストの分析（意図分析）

#### 2.1 リクエストの明確さ

- **明確**：具体的で、明確に定義され、実行可能
- **曖昧**：一般的で、不明確で、明確化が必要
- **不完全**：重要な情報が欠けている

#### 2.2 リクエストのタイプ

- **新機能**：新しい機能の追加
- **バグ修正**：既存の問題の修正
- **リファクタリング**：コード構造の改善
- **アップグレード**：依存関係またはフレームワークの更新
- **マイグレーション**：異なる技術への移行
- **機能強化**：既存機能の改善
- **新規プロジェクト**：最初から始める

#### 2.3 初期スコープ見積もり

- **単一ファイル**：1つのファイルへの変更
- **単一コンポーネント**：1つのコンポーネント/パッケージへの変更
- **複数コンポーネント**：複数のコンポーネントにわたる変更
- **システム全体**：システム全体に影響する変更
- **クロスシステム**：複数のシステムに影響する変更

#### 2.4 初期複雑さ見積もり

- **些細**：シンプルで直接的な変更
- **シンプル**：明確な実装パス
- **普通**：ある程度の複雑さ、複数の考慮事項
- **複雑**：重大な複雑さ、多くの考慮事項

### ステップ3：要件深度の決定

**リクエスト分析に基づいて深度を決定する：**

**最小限の深度** - 以下の場合に使用：

- リクエストが明確でシンプル
- 詳細な要件が不要
- 基本的な理解を文書化するだけ

**標準的な深度** - 以下の場合に使用：

- リクエストに明確化が必要
- 機能的および非機能的要件が必要
- 通常の複雑さ

**包括的な深度** - 以下の場合に使用：

- 複数のステークホルダーを持つ複雑なプロジェクト
- 高リスクまたはクリティカルなシステム
- トレーサビリティを含む詳細な要件が必要

### ステップ4：現在の要件の評価

ユーザーが提供したものを分析する：

- 意図の記述または説明（すでにaudit.mdにログ記録済み）
- 既存の要件ドキュメント（言及された場合はワークスペースを検索する）
- 貼り付けられたコンテンツまたはファイル参照
- Markdown形式以外のドキュメントをMarkdown形式に変換する

### ステップ5：徹底的な完全性分析

**重要**：包括的な分析を使用して要件の完全性を評価する。曖昧さや欠けている詳細がある場合はデフォルトで質問する。

**必須**：以下のすべての領域を評価し、不明確な領域については質問する：

- **機能要件**：コア機能、ユーザーインタラクション、システムの動作
- **非機能要件**：パフォーマンス、セキュリティ、スケーラビリティ、使いやすさ
- **ユーザーシナリオ**：ユースケース、ユーザージャーニー、エッジケース、エラーシナリオ
- **ビジネスコンテキスト**：目標、制約、成功基準、ステークホルダーのニーズ
- **技術コンテキスト**：統合ポイント、データ要件、システム境界
- **品質属性**：信頼性、保守性、テスト可能性、アクセシビリティ

**迷ったら質問する** - 不完全な要件は実装の悪化につながる。

### ステップ5.1：拡張機能のオプトインプロンプト

**必須**：（ワークフロー開始時に`extensions/`サブディレクトリから読み込まれた）読み込まれたすべての`*.opt-in.md`ファイルで`## Opt-In Prompt`セクションをスキャンする。それを宣言する各拡張機能について、ステップ6で作成される明確化質問ファイルにその質問を含める。

回答を受け取った後：

1. `aidlc-docs/aidlc-state.md`の`## Extension Configuration`に各拡張機能の有効化ステータスを記録する：

```markdown
## 拡張機能の設定
| 拡張機能 | 有効 | 決定日 |
|---|---|---|
| [拡張機能名] | [はい/いいえ] | 要件分析 |
```

1. **遅延ルール読み込み**：ユーザーがオプトインした各拡張機能に対して、今すぐ完全なルールファイルを読み込む。ルールファイルは命名規則によって導出される：opt-inファイル名から`.opt-in.md`を削除して`.md`を追加する（例：`security-baseline.opt-in.md` → `security-baseline.md`）。ユーザーがオプトアウトした拡張機能については、完全なルールファイルを読み込まない。

### ステップ6：明確化質問の生成（積極的アプローチ）

- 要件が例外的に明確で完全でない限り、**常に**`aidlc-docs/inception/requirements/requirement-verification-questions.md`を作成する
- 欠けている、不明確な、または曖昧な領域について質問する
- 機能要件、非機能要件、ユーザーシナリオ、ビジネスコンテキストに焦点を当てる
- 質問ドキュメント内のすべての[Answer]:タグに直接記入するようユーザーに依頼する
- 回答に複数選択肢のオプションを提示する場合：
  - オプションにA、B、C、Dなどのラベルを付ける
  - オプションが相互排他的で重複しないようにする
  - カスタム回答のオプションを常に含める：「X) その他（以下の[Answer]:タグの後に記述してください）」
- ドキュメント内のユーザーの回答を待つ
- **必須**：すべての回答の曖昧さを分析し、必要に応じてフォローアップ質問を作成する
- **必須**：すべての曖昧さが解決されるか、ユーザーが明示的に続行を求めるまで質問し続ける

### ⛔ ゲート：ユーザーの回答を待つ

requirement-verification-questions.mdのすべての質問が回答され検証されるまで、ステップ7に進まないこと。
質問ファイルをユーザーに提示して停止する。

### ステップ7：要件ドキュメントの生成

- **前提条件**：ステップ6のゲートを通過していること——すべての回答を受け取り分析済みであること
- `aidlc-docs/inception/requirements/requirements.md`を作成する
- 冒頭に意図分析サマリーを含める：
  - ユーザーリクエスト
  - リクエストタイプ
  - スコープ見積もり
  - 複雑さ見積もり
- 機能要件と非機能要件の両方を含める
- 明確化質問へのユーザーの回答を組み込む
- 主要な要件の簡単なサマリーを提供する

### ステップ8：状態トラッキングの更新

`aidlc-docs/aidlc-state.md`を更新する：

```markdown
## ステージ進捗
### 🔵 INCEPTION PHASE
- [x] ワークスペース検出
- [x] リバースエンジニアリング（該当する場合）
- [x] 要件分析
```

### ステップ9：ログ記録して進む

- タイムスタンプ付きで`aidlc-docs/audit.md`に承認プロンプトをログ記録する
- 以下の構造で完了メッセージを提示する：
     1. **完了アナウンス**（必須）：常にこれで始める：

```markdown
# 🔍 要件分析完了
```

     2. **AIサマリー**（任意）：要件の構造化された箇条書きサマリーを提供する
        - フォーマット：「要件分析で[プロジェクトタイプ/複雑さ]が特定されました：」
        - 主要な機能要件をリスト（箇条書き）
        - 主要な非機能要件をリスト（箇条書き）
        - 関連する場合はアーキテクチャの考慮事項や技術的決定を言及する
        - ワークフロー指示を含めない（「レビューしてください」、「お知らせください」、「次のフェーズに進む」、「進む前に」）
        - 事実に基づいてコンテンツに焦点を当てる
     3. **フォーマットされたワークフローメッセージ**（必須）：常にこの正確なフォーマットで終わる：

```markdown
> **📋 <u>**レビュー必須：**</u>**
> 要件ドキュメントを確認してください：`aidlc-docs/inception/requirements/requirements.md`



> **🚀 <u>**次は何をしますか？**</u>**
>
> **選択してください：**
>
> 🔧 **変更のリクエスト** - レビューに基づいて要件の修正をリクエストする
> [ユーザーストーリーをスキップする場合、このオプションを追加する：]
> 📝 **ユーザーストーリーを追加する** - **ユーザーストーリー**ステージを含める（プロジェクトのシンプルさに基づいて現在スキップされています）
> ✅ **承認して続ける** - 要件を承認して**[ユーザーストーリー/ワークフロープランニング]**に進む

---
```

**注**：ユーザーストーリーステージをスキップする場合のみ「ユーザーストーリーを追加する」オプションを含める。[ユーザーストーリー/ワークフロープランニング]を実際の次のステージ名に置き換える。

- 続行する前に明示的なユーザーの承認を待つ
- タイムスタンプ付きで承認回答を記録する
- aidlc-state.mdで要件分析ステージを完了としてマークする
