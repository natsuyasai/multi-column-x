# アプリケーション設計 - 詳細ステップ

## 目的

**高レベルのコンポーネント識別とサービスレイヤー設計**

アプリケーション設計の焦点：

- 主要な機能コンポーネントとその責任の識別
- コンポーネントインターフェースの定義（詳細なビジネスロジックではない）
- オーケストレーション用のサービスレイヤーの設計
- コンポーネントの依存関係と通信パターンの確立

**注**：詳細なビジネスロジック設計は後の機能設計（ユニットごと、CONSTRUCTIONフェーズ）で行われる

## 前提条件

- コンテキスト評価が完了していること
- 要件評価を推奨（機能的コンテキストを提供する）
- ストーリー開発を推奨（ユーザーストーリーが設計の決定を導く）
- 実行計画がアプリケーション設計ステージを実行すべきことを示していること

## ステップバイステップの実行

### 1. コンテキストの分析

- `aidlc-docs/inception/requirements/requirements.md`と`aidlc-docs/inception/user-stories/stories.md`を読み込む
- 主要なビジネス機能と機能領域を特定する
- 設計スコープと複雑さを決定する

### 2. アプリケーション設計計画の作成

- アプリケーション設計のためのチェックボックス[]付きの計画を生成する
- コンポーネント、責任、メソッド、ビジネスルール、サービスに焦点を当てる
- 各ステップとサブステップにはチェックボックス[]を含める

### 3. 必須設計成果物を計画に含める

- **常に**これらの必須成果物を設計計画に含める：
  - [ ] コンポーネント定義と高レベルの責任を含むcomponents.mdを生成する
  - [ ] メソッドシグネチャを含むcomponent-methods.mdを生成する（ビジネスルールの詳細は後の機能設計で行う）
  - [ ] サービス定義とオーケストレーションパターンを含むservices.mdを生成する
  - [ ] 依存関係の関係と通信パターンを含むcomponent-dependency.mdを生成する
  - [ ] 設計の完全性と一貫性を検証する

### 4. コンテキストに適した質問の生成

**指示**：要件とストーリーを分析して、このアプリケーション設計に関連する質問のみを生成する。以下のカテゴリーを必須チェックリストではなくインスピレーションとして使用する。該当しないカテゴリーはスキップする。

- [Answer]:タグ形式を使用して質問を埋め込む
- このコンテキスト固有の曖昧さや欠けている情報に焦点を当てる
- 設計の決定にユーザー入力が必要な箇所にのみ質問を生成する

**質問カテゴリーの例**（必要に応じて適応する）：

- **コンポーネントの識別** - コンポーネントの境界や組織が不明確な場合のみ
- **コンポーネントのメソッド** - メソッドのシグネチャに明確化が必要な場合のみ（詳細なビジネスルールは後で行う）
- **サービスレイヤーの設計** - サービスのオーケストレーションや境界が曖昧な場合のみ
- **コンポーネントの依存関係** - 通信パターンや依存関係の管理が不明確な場合のみ
- **設計パターン** - アーキテクチャスタイルやパターンの選択にユーザー入力が必要な場合のみ

### 5. アプリケーション設計計画の保存

- `aidlc-docs/inception/plans/application-design-plan.md`として保存する
- ユーザー入力のためのすべての[Answer]:タグを含める
- 計画がすべての設計面をカバーしていることを確認する

### 6. ユーザー入力のリクエスト

- 計画ドキュメントの[Answer]:タグに直接記入するようユーザーに依頼する
- 設計の決定の重要性を強調する

### 7. 回答の収集

- ドキュメント内の[Answer]:タグを使用してすべての質問への回答を提供するまで待つ
- すべての[Answer]:タグが完了するまで進まない

### 8. 回答の分析（必須）

先に進む前に、すべてのユーザーの回答を注意深く確認する：

- **曖昧または不明確な回答**：「mix of」、「somewhere between」、「not sure」、「depends」
- **未定義の基準や用語**：明確な定義のない概念への参照
- **矛盾した回答**：互いに矛盾する回答
- **不足している設計の詳細**：具体的なガイダンスを欠く回答
- **オプションを組み合わせた回答**：明確な決定ルールなしに異なるアプローチを統合する回答

### 9. 必須のフォローアップ質問

ステップ8の分析が曖昧な回答を明らかにした場合、以下が必要です：
- すべての曖昧さが解決されるまで承認に進まない
- 必要なフォローアップの例：
  - 「「AとBの混合」と述べましたが、AとBのどちらを使うかを決定する具体的な基準は何ですか？」
  - 「「AとBの間のどこか」とおっしゃいましたが、正確な中間アプローチを定義できますか？」
  - 「「わからない」と示しましたが、決定に役立つ追加情報は何ですか？」
  - 「「複雑さによって異なる」と述べましたが、どのように複雑さのレベルを定義しますか？」

### 10. アプリケーション設計成果物の生成

- 承認された計画を実行して設計成果物を生成する
- `aidlc-docs/inception/application-design/components.md`を作成する（以下を含む）：
  - コンポーネント名と目的
  - コンポーネントの責任
  - コンポーネントインターフェース
- `aidlc-docs/inception/application-design/component-methods.md`を作成する（以下を含む）：
  - 各コンポーネントのメソッドシグネチャ
  - 各メソッドの高レベルの目的
  - 入力/出力タイプ
  - 注：詳細なビジネスルールは後の機能設計（ユニットごと、CONSTRUCTIONフェーズ）で定義される
- `aidlc-docs/inception/application-design/services.md`を作成する（以下を含む）：
  - サービスの定義
  - サービスの責任
  - サービスのインタラクションとオーケストレーション
- `aidlc-docs/inception/application-design/component-dependency.md`を作成する（以下を含む）：
  - 関係を示す依存関係マトリックス
  - コンポーネント間の通信パターン
  - データフロー図
- 上記で作成された複数の設計ドキュメントを1つの文書に統合した`aidlc-docs/inception/application-design/application-design.md`を作成する。

### 11. 承認のログ記録

- タイムスタンプ付きで`aidlc-docs/audit.md`に承認プロンプトをログ記録する
- 完全な承認プロンプトテキストを含める
- ISO 8601タイムスタンプ形式を使用する

### 12. 完了メッセージの提示

```markdown
# 🏗️ アプリケーション設計完了

[作成されたアプリケーション設計成果物の箇条書きのAI生成サマリー]

> **📋 <u>**レビュー必須：**</u>**
> アプリケーション設計成果物を確認してください：`aidlc-docs/inception/application-design/`

> **🚀 <u>**次は何をしますか？**</u>**
>
> **選択してください：**
>
> 🔧 **変更のリクエスト** - 必要に応じてアプリケーション設計の修正をリクエストする
> [ユニット生成をスキップする場合：]
> 📝 **ユニット生成を追加する** - **ユニット生成**ステージを含める（現在スキップされています）
> ✅ **承認して続ける** - 設計を承認して**[ユニット生成/CONSTRUCTIONフェーズ]**に進む
```

### 13. 明示的な承認の待機

- ユーザーがアプリケーション設計を明示的に承認するまで進まない
- 承認は明確で曖昧さがないこと
- ユーザーが変更をリクエストした場合、設計を更新して承認プロセスを繰り返す

### 14. 承認回答の記録

- タイムスタンプ付きでユーザーの承認回答を`aidlc-docs/audit.md`にログ記録する
- 正確なユーザー回答テキストを含める
- 承認ステータスを明確にマークする

### 15. 進捗の更新

- `aidlc-docs/aidlc-state.md`でアプリケーション設計ステージを完了としてマークする
- 「現在のステータス」セクションを更新する
- 次のステージへの移行を準備する
