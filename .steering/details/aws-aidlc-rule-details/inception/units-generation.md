# ユニット生成 - 詳細ステップ

## 概要

このステージでは2つの統合されたパートを通じてシステムを管理可能な作業単位に分解する：

- **パート1 - プランニング**：質問を含む分解計画の作成、回答の収集、曖昧さの分析、承認の取得
- **パート2 - 生成**：承認された計画を実行してユニット成果物を生成する

**定義**：作業単位は開発目的のユーザーストーリーの論理的なグループ化です。マイクロサービスでは、各ユニットは独立してデプロイ可能なサービスになります。モノリスでは、単一のユニットは論理的なモジュールを持つアプリケーション全体を表します。

**用語**：独立してデプロイ可能なコンポーネントには「サービス」を使用し、サービス内の論理的なグループ化には「モジュール」を使用し、計画のコンテキストには「作業単位」を使用する。

## 前提条件

- コンテキスト評価が完了していること
- 要件評価を推奨（機能的スコープを提供する）
- ストーリー開発を推奨（ストーリーをユニットにマップする）
- アプリケーション設計フェーズが**必須**（コンポーネント、メソッド、サービスを決定する）
- 実行計画が設計フェーズを実行すべきことを示していること

# パート1：プランニング

## ステップ1：作業単位計画の作成

- システムを作業単位に分解するためのチェックボックス[]付きの計画を生成する
- システムを管理可能な開発ユニットに分解することに焦点を当てる
- 各ステップとサブステップにはチェックボックス[]を含める

## ステップ2：必須ユニット成果物を計画に含める

計画に**常に**これらの必須成果物を含める：

- [ ] ユニット定義と責任を含む`aidlc-docs/inception/application-design/unit-of-work.md`を生成する
- [ ] 依存関係マトリックスを含む`aidlc-docs/inception/application-design/unit-of-work-dependency.md`を生成する
- [ ] ストーリーをユニットにマッピングする`aidlc-docs/inception/application-design/unit-of-work-story-map.md`を生成する
- [ ] **グリーンフィールドのみ**：コード組織戦略を`unit-of-work.md`に文書化する（構造パターンについてはcode-generation.mdを参照）
- [ ] ユニットの境界と依存関係を検証する
- [ ] すべてのストーリーがユニットに割り当てられていることを確認する

## ステップ3：コンテキストに適した質問の生成

**指示**：要件、ストーリー、アプリケーション設計を分析して、この分解問題に関連する質問のみを生成する。以下のカテゴリーを必須チェックリストではなくインスピレーションとして使用する。該当しないカテゴリーはスキップする。

- [Answer]:タグ形式を使用して質問を埋め込む
- このコンテキスト固有の曖昧さや欠けている情報に焦点を当てる
- 意思決定にユーザー入力が必要な箇所にのみ質問を生成する

**質問カテゴリーの例**（必要に応じて適応する）：

- **ストーリーのグループ化** - 複数のストーリーが存在してグループ化戦略が不明確な場合のみ
- **依存関係** - 複数のユニットが予想され統合アプローチが曖昧な場合のみ
- **チームの整合性** - チーム構造や所有権が不明確な場合のみ
- **技術的考慮事項** - スケーラビリティ/デプロイ要件がユニット間で異なる場合のみ
- **ビジネスドメイン** - ドメイン境界や境界コンテキストが不明確な場合のみ
- **コードの組織（グリーンフィールドのマルチユニットのみ）** - デプロイメントモデルとディレクトリ構造の好みを確認する

## ステップ4：UOW計画の保存

- `aidlc-docs/inception/plans/unit-of-work-plan.md`として保存する
- ユーザー入力のためのすべての[Answer]:タグを含める
- 計画がシステム分解のすべての側面をカバーしていることを確認する

## ステップ5：ユーザー入力のリクエスト

- 計画ドキュメントの[Answer]:タグに直接記入するようユーザーに依頼する
- 分解の決定の重要性を強調する

## ステップ6：回答の収集

- ドキュメント内の[Answer]:タグを使用してすべての質問への回答を提供するまで待つ
- すべての[Answer]:タグが完了するまで進まない

## ステップ7：回答の分析（必須）

先に進む前に、すべてのユーザーの回答を注意深く確認する：

- **曖昧または不明確な回答**：「mix of」、「somewhere between」、「not sure」、「depends」
- **未定義の基準や用語**：明確な定義のない概念への参照
- **矛盾した回答**：互いに矛盾する回答
- **不足している生成の詳細**：具体的なガイダンスを欠く回答
- **オプションを組み合わせた回答**：明確な決定ルールなしに異なるアプローチを統合する回答

## ステップ8：必須のフォローアップ質問

ステップ7の分析が曖昧な回答を明らかにした場合、以下が必要です：
- すべての曖昧さが解決されるまで承認に進まない
- 必要なフォローアップの例：
  - 「「AとBの混合」と述べましたが、AとBのどちらを使うかを決定する具体的な基準は何ですか？」
  - 「「AとBの間のどこか」とおっしゃいましたが、正確な中間アプローチを定義できますか？」
  - 「「わからない」と示しましたが、決定に役立つ追加情報は何ですか？」
  - 「「複雑さによって異なる」と述べましたが、どのように複雑さのレベルを定義しますか？」

## ステップ9：承認のリクエスト

- 「**作業単位計画が完了しました。aidlc-docs/inception/plans/unit-of-work-plan.mdの計画を確認してください。生成に進む準備ができていますか？**」と確認する
- ユーザーが確認するまで進まない

## ステップ10：承認のログ記録

- タイムスタンプ付きでaudit.mdにプロンプトと回答をログ記録する
- ISO 8601タイムスタンプ形式を使用する
- 完全な承認プロンプトテキストを含める

## ステップ11：進捗の更新

- aidlc-state.mdでユニットプランニングを完了としてマークする
- 「現在のステータス」セクションを更新する
- ユニット生成への移行を準備する

# パート2：生成

## ステップ12：作業単位計画の読み込み

- [ ] `aidlc-docs/inception/plans/unit-of-work-plan.md`から完全な計画を読み込む
- [ ] 次の未完了ステップを特定する（最初の[ ]チェックボックス）
- [ ] そのステップのコンテキストと要件を読み込む

## ステップ13：現在のステップの実行

- [ ] 現在のステップが説明する内容を正確に実行する
- [ ] 計画で指定されたユニット成果物を生成する
- [ ] プランニングから承認された分解アプローチに従う
- [ ] 計画で指定された基準と境界を使用する

## ステップ14：進捗の更新

- [ ] 作業単位計画で完了したステップを[x]としてマークする
- [ ] `aidlc-docs/aidlc-state.md`の現在のステータスを更新する
- [ ] 生成されたすべての成果物を保存する

## ステップ15：継続または完了

- [ ] 残りのステップがある場合はステップ12に戻る
- [ ] すべてのステップが完了した場合は設計ステージのためにユニットが準備できていることを確認する
- [ ] ユニット生成ステージを完了としてマークする

## ステップ16：完了メッセージの提示

```markdown
# 🔧 ユニット生成完了

[作成されたユニットと分解の箇条書きのAI生成サマリー]

> **📋 <u>**レビュー必須：**</u>**
> ユニット生成成果物を確認してください：`aidlc-docs/inception/application-design/`

> **🚀 <u>**次は何をしますか？**</u>**
>
> **選択してください：**
>
> 🔧 **変更のリクエスト** - 必要に応じてユニット生成の修正をリクエストする
> ✅ **承認して続ける** - ユニットを承認して**CONSTRUCTIONフェーズ**に進む
```

## ステップ17：明示的な承認の待機

- ユーザーがユニット生成を明示的に承認するまで進まない
- 承認は明確で曖昧さがないこと
- ユーザーが変更をリクエストした場合、ユニットを更新して承認プロセスを繰り返す

## ステップ18：承認回答の記録

- タイムスタンプ付きでユーザーの承認回答を`aidlc-docs/audit.md`にログ記録する
- 正確なユーザー回答テキストを含める
- 承認ステータスを明確にマークする

## ステップ19：進捗の更新

- `aidlc-docs/aidlc-state.md`でユニット生成ステージを完了としてマークする
- 「現在のステータス」セクションを更新する
- CONSTRUCTIONフェーズへの移行を準備する

## 重要ルール

### プランニングフェーズのルール

- コンテキストに関連する質問のみを生成する
- すべての質問に[Answer]:タグ形式を使用する
- 進む前にすべての回答の曖昧さを分析する
- フォローアップ質問ですべての曖昧さを解決する
- 生成前にユーザーの明示的な承認を得る

### 生成フェーズのルール

- **ハードコードしたロジックは使用しない**：作業単位計画に記述されたことのみを実行する
- **計画に厳密に従う**：ステップシーケンスから逸脱しない
- **チェックボックスを更新する**：各ステップ完了直後に[x]をマークする
- **承認されたアプローチを使用する**：プランニングからの分解方法論に従う
- **完了を確認する**：進む前にすべてのユニット成果物が完全であることを確認する

## 完了基準

- すべてのプランニング質問に回答し曖昧さが解決されている
- 計画に対してユーザーの承認が得られている
- 作業単位計画のすべてのステップが[x]でマークされている
- 計画に従って、すべてのユニット成果物が生成されている：
  - ユニット定義を含む`unit-of-work.md`
  - 依存関係マトリックスを含む`unit-of-work-dependency.md`
  - ストーリーマッピングを含む`unit-of-work-story-map.md`
- ユニットが確認されてユニットごとの設計ステージの準備ができている
