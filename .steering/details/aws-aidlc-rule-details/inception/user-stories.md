# ユーザーストーリー - 詳細ステップ

## 目的

**要件をユーザー中心のストーリーと受け入れ基準に変換する**

ユーザーストーリーの焦点：

- ビジネス要件をユーザー中心のナラティブに変換する
- 各ストーリーに明確な受け入れ基準を定義する
- 異なるステークホルダータイプを表すユーザーペルソナを作成する
- チーム間の共通理解を確立する
- 実装のためのテスト可能な仕様を提供する

## 前提条件

- ワークスペース検出が完了していること
- 要件分析を推奨（利用可能な場合は要件を参照できる）
- ワークフロープランニングがユーザーストーリーステージを実行すべきことを示していること

## インテリジェント評価ガイドライン

**ユーザーストーリーを実行するタイミング**：進む前にこの拡張評価を使用する：

### 高優先度実行（常に実行）

- **新しいユーザー機能**：ユーザーが直接インタラクトする新しい機能
- **ユーザーエクスペリエンスの変更**：既存のユーザーワークフローまたはインターフェースへの変更
- **マルチペルソナシステム**：異なるタイプのユーザーにサービスを提供するアプリケーション
- **顧客向けAPI**：外部ユーザーまたはシステムが使用するサービス
- **複雑なビジネスロジック**：複数のシナリオまたはビジネスルールを持つ要件
- **クロスチームプロジェクト**：複数のチーム間で共通理解が必要な作業

### 中優先度実行（複雑さを評価する）

- **バックエンドのユーザー影響**：ユーザーエクスペリエンスに間接的に影響する内部変更
- **パフォーマンス改善**：ユーザーに見えるメリットのある機能強化
- **統合作業**：ユーザーワークフローに影響するシステム接続
- **データ変更**：ユーザーデータ、レポート、または分析に影響する変更
- **セキュリティ強化**：ユーザー認証またはアクセス許可に影響する変更

### 複雑さ評価の要素

中優先度のケースでは、以下のいずれかが当てはまる場合にユーザーストーリーを実行する：

- **スコープ**：変更が複数のコンポーネントまたはユーザータッチポイントにまたがる
- **曖昧さ**：ストーリーが明確にできる不明確な側面が要件にある
- **リスク**：ビジネスへの影響が大きいか、誤解の可能性がある
- **ステークホルダー**：複数のビジネスステークホルダーが要件に関与している
- **テスト**：ユーザー受け入れテストが必要になる
- **オプション**：複数の有効な実装アプローチが存在する

### 単純なケースのみスキップ

- **純粋なリファクタリング**：ユーザーへの影響がゼロの内部コード改善
- **孤立したバグ修正**：明確なスコープを持つシンプルで明確な修正
- **インフラのみ**：ユーザー向けの効果がない変更
- **開発者ツール**：ビルドプロセス、CI/CD、または開発環境の変更
- **ドキュメント**：機能に影響しない更新

### デフォルトの判断ルール

**迷ったら、ユーザーストーリーを含めて明確化の質問をする。** 適切な明確化を伴う包括的なストーリーを作成するオーバーヘッドは、通常以下のメリットによって相殺される：

- より明確な要件理解
- より良いチームの整合性
- 改善されたテスト基準
- 強化されたステークホルダーとのコミュニケーション
- 実装リスクの軽減
- 開発中のコスト高な変更の減少
- より良いユーザーエクスペリエンスの結果

# パート1：プランニング

## ステップ1：ユーザーストーリーの必要性の検証（必須）

**重要**：ユーザーストーリーに進む前に、この評価を実施する：

### 評価プロセス

1. **リクエストのコンテキストを分析する**：
   - 元のユーザーリクエストと要件を確認する
   - ユーザー向けの変更と内部のみの変更を特定する
   - 作業の複雑さとスコープを評価する
   - ビジネスステークホルダーの関与を評価する

2. **評価基準を適用する**：
   - 高優先度の指標をチェックする（常に実行）
   - 中優先度の要素を評価する（複雑さに基づく判断）
   - スキップすべき単純なケースではないことを確認する

3. **評価の判断を文書化する**：
   - `aidlc-docs/inception/plans/user-stories-assessment.md`を作成する
   - このリクエストにユーザーストーリーが有用な理由を含める
   - 適用される具体的な評価基準を参照する
   - 期待されるメリット（明確さ、テスト、ステークホルダーの整合）を説明する

4. **正当化される場合のみ進む**：
   - ユーザーストーリーはプロジェクトに明確な価値をもたらす必要がある
   - 評価は具体的なメリットがオーバーヘッドを上回ることを示す必要がある
   - 判断はプロジェクトのステークホルダーに対して説明できるものであること

### 評価ドキュメントのテンプレート

```markdown
# ユーザーストーリー評価

## リクエスト分析
- **元のリクエスト**：[簡単なサマリー]
- **ユーザーへの影響**：[直接/間接/なし]
- **複雑さレベル**：[シンプル/中/複雑]
- **ステークホルダー**：[関係者のリスト]

## 満たした評価基準
- [ ] 高優先度：[適用される基準のリスト]
- [ ] 中優先度：[複雑さの正当化を含む適用される基準のリスト]
- [ ] メリット：[ユーザーストーリーから期待される価値]

## 判断
**ユーザーストーリーを実行する**：[はい/いいえ]
**理由**：[詳細な正当化]

## 期待される結果
- [ユーザーストーリーが提供する具体的なメリットのリスト]
- [ストーリーがプロジェクトの成功をどのように改善するか]
```

## ステップ2：ストーリー計画の作成

- プロダクトオーナーとしての役割を引き受ける
- ストーリー開発のためのステップバイステップの実行チェックリストを含む包括的な計画を生成する
- 各ステップとサブステップにはチェックボックス[]を含める
- 要件をユーザーストーリーに変換するための方法論とアプローチに焦点を当てる

## ステップ3：コンテキストに適した質問の生成

**指示**：要件とコンテキストを徹底的に分析して、明確化がストーリーの品質とチームの理解を向上させるすべての領域を特定する。包括的なユーザーストーリー開発を確保するために積極的に質問する。

**重要**：ストーリーの品質に影響する可能性のある曖昧さや欠けている詳細がある場合は、デフォルトで質問する。不完全または不明確なストーリーを作成するよりも、多くの質問をする方が良い。

**質問のフォーマットルールについては`common/question-format-guide.md`を参照してください**

- [Answer]:タグ形式を使用して質問を埋め込む
- 曖昧さ、欠けている情報、または明確化が必要な領域に焦点を当てる
- ユーザー入力がストーリー作成の決定を改善できる箇所に質問を生成する
- **迷ったら質問する** - 過信は質の低いストーリーにつながる

**評価すべき質問カテゴリー**（すべてのカテゴリーを検討する）：

- **ユーザーペルソナ** - ユーザーのタイプ、役割、特性、動機について質問する
- **ストーリーの粒度** - 適切な詳細レベル、ストーリーのサイズ、分解アプローチについて質問する
- **ストーリーのフォーマット** - フォーマットの好み、テンプレートの使用、ドキュメント標準について質問する
- **分解アプローチ** - 組織化方法、優先順位付け、グループ化戦略について質問する
- **受け入れ基準** - 詳細レベル、フォーマット、テストアプローチ、検証方法について質問する
- **ユーザージャーニー** - ユーザーワークフロー、インタラクションパターン、エクスペリエンスフローについて質問する
- **ビジネスコンテキスト** - ビジネス目標、成功指標、ステークホルダーのニーズについて質問する
- **技術的制約** - 技術的制限、統合要件、システム境界について質問する

## ステップ4：必須ストーリー成果物を計画に含める

- **常に**これらの必須成果物をストーリー計画に含める：
  - [ ] INVEST基準に従ったユーザーストーリーを含むstories.mdを生成する
  - [ ] ユーザーアーキタイプと特性を含むpersonas.mdを生成する
  - [ ] ストーリーが独立（Independent）、交渉可能（Negotiable）、価値がある（Valuable）、見積もり可能（Estimable）、小さい（Small）、テスト可能（Testable）であることを確認する
  - [ ] 各ストーリーに受け入れ基準を含める
  - [ ] 関連するユーザーストーリーにペルソナをマップする

## ステップ5：ストーリーオプションの提示

- 計画ドキュメントにストーリー分解の異なるアプローチを含める：
  - **ユーザージャーニーベース**：ユーザーワークフローとインタラクションに沿ったストーリー
  - **機能ベース**：システムの機能と能力を中心に整理されたストーリー
  - **ペルソナベース**：異なるユーザータイプとそのニーズでグループ化されたストーリー
  - **ドメインベース**：ビジネスドメインまたはコンテキストを中心に整理されたストーリー
  - **エピックベース**：サブストーリーを含む階層的なエピックとして構造化されたストーリー
- 各アプローチのトレードオフとメリットを説明する
- 明確な判断基準を持つハイブリッドアプローチを可能にする

## ステップ6：ストーリー計画の保存
- ファイル名：`story-generation-plan.md`
- ユーザー入力のためのすべての[Answer]:タグを含める
- 計画がすべてのストーリー開発の側面をカバーしていることを確認する

## ステップ7：ユーザー入力のリクエスト

- ストーリー計画ドキュメントのすべての[Answer]:タグに直接記入するようユーザーに依頼する
- 監査証跡と意思決定ドキュメントの重要性を強調する
- 進む前にすべての質問に回答する必要があることを説明する

## ステップ8：回答の収集

- ドキュメント内の[Answer]:タグを使用してすべての質問への回答を提供するまで待つ
- すべての[Answer]:タグが完了するまで進まない

## ステップ9：回答の分析（必須）

進む前に、すべてのユーザーの回答について以下を注意深く確認する：

- **曖昧または不明確な回答**：「mix of」、「somewhere between」、「not sure」、「depends」、「maybe」、「probably」
- **未定義の基準や用語**：明確な定義のない概念への参照
- **矛盾した回答**：互いに矛盾する回答
- **不足している生成の詳細**：具体的なガイダンスを欠く回答
- **オプションを組み合わせた回答**：明確な決定ルールなしに異なるアプローチを統合する回答
- **不完全な説明**：外部要素を定義せずに参照する回答
- **前提に基づく回答**：明示的に述べられていない知識を前提とする回答

## ステップ10：必須のフォローアップ質問

ステップ9の分析が曖昧な回答を明らかにした場合、以下が必要です：
- すべての曖昧さが完全に解決されるまで承認に進まない
- **重要**：徹底的に行う - 不明確な回答ごとにフォローアップ質問をする
- 必要なフォローアップの例：
  - 「「AとBの混合」と述べましたが、AとBのどちらを使うかを決定する具体的な基準は何ですか？」
  - 「「AとBの間のどこか」とおっしゃいましたが、正確な中間アプローチを定義できますか？」
  - 「「わからない」と示しましたが、決定に役立つ追加情報は何ですか？」
  - 「「複雑さによって異なる」と述べましたが、どのように複雑さのレベルと閾値を定義しますか？」
  - 「「ハイブリッドアプローチ」を選択しましたが、各方法を使用する具体的なルールは何ですか？」
  - 「「おそらくX」とおっしゃいましたが、確実にXになる要因とXにならない要因は何ですか？」
  - 「「標準的な慣行」に言及しましたが、その標準的な慣行を定義できますか？」

## ステップ11：実装の詳細を避ける

- 優先順位付けや開発タスクではなく、ストーリー作成の方法論に焦点を当てる
- この段階では技術的な生成について議論しない
- 開発タイムラインやスプリントプランニングの作成を避ける
- ストーリーの構造とフォーマットの決定に焦点を当てる

## ステップ12：承認プロンプトのログ記録

- 承認を求める前に、タイムスタンプ付きでプロンプトを`aidlc-docs/audit.md`にログ記録する
- 完全な承認プロンプトテキストを含める
- ISO 8601タイムスタンプ形式を使用する

## ステップ13：計画の明示的な承認の待機

- ユーザーがストーリーアプローチを明示的に承認するまで進まない
- 承認は明確で曖昧さがないこと
- ユーザーが変更をリクエストした場合、計画を更新して承認プロセスを繰り返す

## ステップ14：承認回答の記録

- タイムスタンプ付きでユーザーの承認回答を`aidlc-docs/audit.md`にログ記録する
- 正確なユーザー回答テキストを含める
- 承認ステータスを明確にマークする

## ステップ20：完了メッセージの提示

- 以下の構造で完了メッセージを提示する：
     1. **完了アナウンス**（必須）：常にこれで始める：

```markdown
# 📚 ユーザーストーリー完了
```

     2. **AIサマリー**（任意）：生成されたストーリーの構造化された箇条書きサマリーを提供する
        - フォーマット：「ユーザーストーリー生成で[説明]を作成しました：」
        - 生成されたキーペルソナをリスト（箇条書き）
        - 作成されたユーザーストーリーを数と整理方法とともにリスト

        - 事実に基づいてコンテンツに焦点を当てる
     3. **フォーマットされたワークフローメッセージ**（必須）：常にこの正確なフォーマットで終わる：

```markdown
> **📋 <u>**レビュー必須：**</u>**
> ユーザーストーリーとペルソナを確認してください：`aidlc-docs/inception/user-stories/stories.md`と`aidlc-docs/inception/user-stories/personas.md`



> **🚀 <u>**次は何をしますか？**</u>**
>
> **選択してください：**
>
> 🔧 **変更のリクエスト** - レビューに基づいてストーリーまたはペルソナの修正をリクエストする
> ✅ **承認して続ける** - ユーザーストーリーを承認して**ワークフロープランニング**に進む

---
```

# 重要ルール

## プランニングフェーズのルール

- **コンテキストに適した質問**：このコンテキストに関連する質問のみを生成する
- **必須の回答分析**：進む前に常に回答の曖昧さを分析する
- **曖昧さがある状態で進まない**：生成前にすべての曖昧な回答を解決する
- **明示的な承認が必要**：生成を開始する前にユーザーが計画を承認する必要がある

## 生成フェーズのルール

- **ハードコードしたロジックは使用しない**：ストーリー生成計画に記述されたことのみを実行する
- **計画に厳密に従う**：ステップシーケンスから逸脱しない
- **チェックボックスを更新する**：各ステップ完了直後に[x]をマークする
- **承認されたアプローチを使用する**：プランニングからのストーリーアプローチに従う
- **完了を確認する**：進む前にすべてのストーリー成果物が完全であることを確認する

## 完了基準

- すべてのプランニング質問に回答し曖昧さが解決されている
- ストーリー計画に対してユーザーの明示的な承認が得られている
- ストーリー生成計画のすべてのステップが[x]でマークされている
- 計画に従って、すべてのストーリー成果物が生成されている（stories.md、personas.md）
- 生成されたストーリーがユーザーに明示的に承認されている
- ストーリーが確認されて次のステージの準備ができている
